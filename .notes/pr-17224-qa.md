## Q1: For `ToolProviderToolInfo` in `packages/core/src/tool-provider/runtime.ts`, did we change the name of the tool slug?

**Short answer:** Yes — but only at *runtime resolution*, not in `ToolProviderToolInfo` itself.

### What stays the same
- `ToolProviderToolInfo.slug` (in `types.ts:37–46`) is the **fully-qualified, natural** tool slug as reported by the provider's `listTools()` (e.g. `gmail.fetch_emails`, `GMAIL_FETCH_EMAILS`).
- That slug is used everywhere we **discover or store** a tool — UI pickers, agent form, `toolProviders[providerId].tools` map.

### What gets renamed (and where)
- In `runtime.ts` (`resolveStoredToolProviders`), when **more than one connection** is bound to the same `(providerId, toolkit)`, each resolved tool's slug gets a `__<SUFFIX>` appended so the LLM can disambiguate between connections:
  - `gmail.fetch_emails` → `gmail.fetch_emails__WORK` and `gmail.fetch_emails__PERSONAL`
- The suffix comes from `buildConnectionSuffix(label, usedSuffixes)` (runtime.ts:34) which:
  - Uppercases the label
  - Replaces non-`[A-Z0-9_]` chars with `_`
  - Collapses repeats, trims, falls back to `'CONN'` if empty
  - Appends `_2`, `_3`, … on collision
- **Single-connection toolkits keep the natural slug** — no suffix is added (runtime.ts:58–59).
- Each renamed tool also gets a routing hint appended to its `description` so the model knows which connection it's calling.

### Why
- The agent's saved `toolProviders[providerId].tools` map stores the **natural** slug + toolkit only — no connection identity baked into the key.
- At runtime we fan out per pinned connection and rename only when there's ambiguity. This keeps stored config stable across rename/disconnect/re-pin without rewriting form state.

### Files to look at
- `packages/core/src/tool-provider/types.ts:37-46` — `ToolProviderToolInfo` (unchanged slug shape)
- `packages/core/src/tool-provider/runtime.ts:34-50` — `buildConnectionSuffix`
- `packages/core/src/tool-provider/runtime.ts:137-162` — suffix application + rename loop

## Q2: Will the editor work as it did before?

**Short answer:** Yes — the legacy `integrationTools` path is untouched and runs in parallel to the new `toolProviders` path.

### How they coexist
In `packages/editor/src/namespaces/agent.ts` `createAgentFromStoredConfig`, the merge order is:

```ts
tools = { ...registryTools, ...mcpTools, ...integrationTools, ...providerTools };
```

- `integrationTools` — **v1 legacy path**, resolved by `resolveStoredIntegrationTools()`. No change in behavior. Still works for Arcade and any other provider that has only `resolveTools` (v1).
- `providerTools` — **new v2 path**, resolved by `resolveStoredToolProviders()` from `@mastra/core/tool-provider`. Connection-aware. Only fires if `storedAgent.toolProviders` is present.

### What this means for existing stored agents
- An agent saved before this PR has `integrationTools: {...}` and no `toolProviders` field — `hasToolProviders` is false, `providerTools = {}`, only the legacy path runs. **Zero behavior change.**
- A new agent saved after this PR can have either field (or both). Both resolve, and `providerTools` wins on key collision (spread order).

### When can the legacy path be removed?
Not yet. Blockers:
- Arcade provider only implements `resolveTools` (v1), no `resolveToolsV2`.
- Existing stored agents in users' databases reference `integrationTools`.
- `rules` on `StorageToolConfig` (v1 conditional variants) is not yet ported to v2.

Realistic timeline: next major.

### Files
- `packages/editor/src/namespaces/agent.ts:621-649` — both paths merged
- `packages/editor/src/namespaces/agent.ts:1095-1112` — `resolveStoredIntegrationTools` (legacy, untouched)

## Q3: For `ListConnectionsOpts` why are we using `cursor`?

**Short answer:** Because the underlying providers (Composio in particular) paginate their `connectedAccounts.list` API with an opaque cursor, and we pass it through unchanged.

### The shape (`types.ts:317-328`)
```ts
export interface ListConnectionsOpts {
  toolkit: string;
  userIds?: string[];
  userId?: string;
  cursor?: string;   // opaque, returned by previous call
  limit?: number;    // adapters clamp to default 50, max 200
}

export interface ListConnectionsResult {
  items: ExistingConnection[];
  nextCursor?: string;  // null/undefined when exhausted
}
```

### Why opaque cursor (not offset/page)
- **Provider-driven**: Composio's `connectedAccounts.list({ cursor })` returns `nextCursor` — it's their token format, not ours. We pass it back round-trip (`composio.ts:342, 355`).
- **No semantics on our side**: The string is opaque — adapters MUST NOT parse, derive, or hash it. They just hand it back on the next call.
- **Stable under inserts**: Cursor-based pagination is more robust than `page=N` against concurrent inserts (no duplicate/skipped rows mid-iteration).
- **Matches the rest of Mastra**: List APIs in core (threads, messages, runs) use the same `{ cursor, limit } → { items, nextCursor }` shape.

### Why it's worth having at all
- One author can accumulate many connections per toolkit (e.g. a customer-support team with one Gmail connection per support inbox).
- Admin cross-author listing (`userIds: [...]`) can fan out to dozens/hundreds of authors. Pagination is required.
- The runtime fan-out (`resolveStoredToolProviders`) does NOT use `listConnections` — it only ever needs pinned IDs from the agent config, so pagination doesn't slow agent boot. `listConnections` is only used by the UI (picker, integrations page).

### Limit clamp
- Default 50, max 200 (`types.ts:326`).
- Composio adapter clamps via `clampLimit` (`composio.ts:337`).
- Prevents a malicious or buggy caller from asking for 100k connections in one call.

### Files
- `packages/core/src/tool-provider/types.ts:317-350` — `ListConnectionsOpts` + `ListConnectionsResult`
- `packages/editor/src/providers/composio.ts:330-356` — passthrough of cursor/nextCursor
- `packages/core/src/tool-provider/base.ts:147` — abstract `listConnections`

## Q4: Do we use `cursor` anywhere else? Usually it's `pagination: { page, perPage, hasMore }`.

**Short answer:** No — `cursor` is an **outlier** in `ListConnectionsOpts`. Everywhere else in core (datasets, experiments, logger, observability, memory, scores) and even **everywhere else in this same PR** (`listTools`, `listToolkits`) uses `{ pagination: { page, perPage, hasMore } }`.

### Audit
- `packages/core/src/tool-provider/types.ts:325, 349` — **the only `cursor`/`nextCursor` in core**.
- `packages/memory/src/tools/om-tools.ts:1271` — internal OM tool, unrelated.
- All other core list APIs: `{ pagination: { page, perPage, hasMore } }`:
  - `dataset.ts`, `manager.ts` — datasets
  - `experiments/...` — experiments
  - `observability/...` — feedback, traces
  - `logger/dual-logger.ts` — logs
  - `memory/base.ts` — `validatePagination(page, perPage)`
- **Same PR, same file** (`tool-provider/types.ts`):
  - `ListToolProviderToolsOptions` (line 51) uses `{ page, perPage }`
  - `ToolProviderListResult<T>` (line 65) returns `{ pagination: { total, page, perPage, hasMore } }`
  - `ListToolkitsV2Opts` (line 207) uses `{ page, perPage }`

So `ListConnectionsOpts` is the **only** place in the PR using a cursor.

### Why it ended up that way
Composio's `connectedAccounts.list` SDK returns an opaque `nextCursor` (their pagination model). The implementer mirrored Composio's shape on our type. It works, but it breaks Mastra convention.

### Verdict
**Recommend changing to `{ page, perPage }` + `{ pagination: { total?, page, perPage, hasMore } }`** for consistency with the rest of the codebase and the rest of this PR.

Adapter cost:
- **Composio**: it does happen to expose a `page` parameter too, not just `cursor` — confirm in SDK types. If only `cursor`, the adapter can internally maintain a `page → cursor` map per request session, or compute offset from `page * perPage`. (Composio SDK 0.6.x supports `page`; the implementer just used `cursor` because it's what the response returns.)
- **libsql/inmemory storage**: trivial — they're SQL/JS arrays, page+offset is native.

### Files to change if we converge
- `packages/core/src/tool-provider/types.ts:317-350` — flip `ListConnectionsOpts` + `ListConnectionsResult`
- `packages/editor/src/providers/composio.ts:330-356` — use `composio.connectedAccounts.list({ page, limit })` instead of `{ cursor, limit }`
- `packages/server/src/server/handlers/tool-providers.ts` LIST_CONNECTIONS — pass page/perPage instead of cursor
- `client-sdks/client-js/src/resources/tool-provider.ts` `listConnections` — signature flip
- `packages/playground/src/domains/tool-providers/hooks/use-existing-connections.ts` — UI doesn't paginate yet, so no caller-side break

### Files
- All cited above + audit results

## Q6: Is `mastra_tool_provider_connections` too generic a table name?

**Short answer:** It's consistent with the existing table naming convention but it IS verbose. The other Mastra tables tend toward shorter, domain-aligned names.

### Convention audit
Looking at neighboring tables in `constants.ts:30-49`:
```
mastra_datasets             (short)
mastra_experiments          (short)
mastra_channel_installations (medium)
mastra_channel_config       (medium)
mastra_workflow_snapshot    (medium)
mastra_tool_provider_connections   ← new, longest
```

So `mastra_tool_provider_connections` is the longest table name in core. Channel tables use the same compound style (`channel_installations`, `channel_config`), so the precedent exists.

### Is it too generic?
**Not really** — it accurately describes the contents:
- It stores OAuth/auth **connections** (not tools, not toolkits)
- These connections belong to **tool providers** (Composio, Arcade, etc.)
- `tool_providers` is the domain name baked into `packages/core/src/tool-provider/`

A shorter name would be misleading:
- `mastra_connections` — too vague (channels also have "connections" semantically; database connections, etc.)
- `mastra_provider_connections` — drops the "tool" disambiguator; "provider" is also overloaded (`processor-providers`, `browser-providers`, `filesystem-providers`)
- `mastra_oauth_connections` — wrong, not all connections are OAuth (could be API key, basic auth)
- `mastra_integrations` — would conflict with the legacy v1 `integrationTools` concept

### Verdict
**Keep as-is**, but worth flagging as a follow-up if the team wants a snappier name. The cost to change is non-trivial — needs adapter migrations across all storage backends (libsql, clickhouse, cloudflare KV, postgres if/when added) and any production data already written.

### Risk
- Once shipped, renaming requires a storage migration in every adapter. **Lock this name in before release.**

### Files
- `packages/core/src/storage/constants.ts:49, 86, 354, 663, 673-674`
- `packages/core/src/storage/domains/tool-provider-connections/base.ts:25`
- `packages/core/src/storage/domains/operations/inmemory.ts:49`

## Q7: Should the tool-provider-connections storage stay in core, or move scoped to editor / agent builder?

**Short answer:** **Leave it in core.** The precedent for editor/builder-only features (channels, MCP clients/servers, agents/workspaces/skills/prompt-blocks) is that they all live in core storage and editor/builder packages consume them. Pulling tool-provider-connections out would break that pattern with no functional gain.

### Precedent — every editor/builder-only table is in core
From `packages/core/src/storage/constants.ts`:
```
mastra_channel_installations    ← Slack/Discord (Builder channels only)
mastra_channel_config            ← Builder channels only
mastra_mcp_clients               ← Builder/editor only
mastra_mcp_servers               ← Builder/editor only
mastra_agents                    ← stored agents (Builder)
mastra_agent_versions
mastra_workspaces                ← Builder workspaces
mastra_workspace_versions
mastra_skills                    ← Builder skills
mastra_skill_versions
mastra_prompt_blocks             ← Builder prompts
mastra_scorer_definitions        ← Builder scorers
mastra_tool_provider_connections ← (new)
```

None of these are used outside the editor/builder code path, yet all of them live in **core**. The pattern is: **core owns the schema, storage adapter implementations, and abstract base class. Editor/server packages consume via `storage.getStore(...)`.**

### Why this works
- **Single storage migration surface.** Every adapter (libsql, clickhouse, cloudflare KV, future postgres) reads one place — `core/src/storage/constants.ts` — to know which tables to provision. Splitting tool-provider tables into `@mastra/editor/storage` would force each adapter to import editor packages or duplicate registration logic.
- **Adapter packages don't depend on editor.** `stores/libsql`, `stores/clickhouse`, `stores/cloudflare` all depend on `@mastra/core`, not on `@mastra/editor`. Moving the schema into editor would force a backwards dep or duplicated constants.
- **Server handlers do `storage.getStore('toolProviderConnections')`** (in `tool-providers.ts:271, 396, 541, 618, 682`). The store factory lives in core's base storage class. That registration would have to move too.
- **No leak risk.** Users who don't enable the editor get an empty table. SQLite/clickhouse `CREATE TABLE IF NOT EXISTS` is cheap.

### What "scope to editor" would actually require
1. Add a separate storage domain registration mechanism (`editor.registerStorageDomain('toolProviderConnections', ...)`)
2. Move `TOOL_PROVIDER_CONNECTIONS_SCHEMA` and `TABLE_TOOL_PROVIDER_CONNECTIONS` into `@mastra/editor`
3. Each adapter (`stores/libsql`, etc.) becomes editor-aware OR registers via callback at init
4. `MastraStorage.getStore<'toolProviderConnections'>` type union needs an extension point
5. Tests + types ripple

This is a **significant refactor with zero behavioral benefit**. The schema is already feature-gated at the consumer layer (only `tool-providers.ts` handler reads it).

### Verdict
**Leave in core.** Matches existing pattern (channels, MCP, stored-agents, workspaces, skills are all editor-only but in core storage). Refactoring to scope it to editor is not worth the work.

### Files
- `packages/core/src/storage/constants.ts:30-86` — all editor/builder-only tables sit here today
- `packages/server/src/server/handlers/tool-providers.ts:271, 396, 541, 618, 682` — consumers via `storage.getStore('toolProviderConnections')`

## Q8: Are the methods in `ToolProviderConnectionsStorage` (`base.ts`) a bit too generic?

**Short answer:** Yes — `get`, `upsert`, `list`, `delete` are bare CRUD verbs and they break the pattern used by sibling domains. Worth tightening.

### Current API
```ts
abstract class ToolProviderConnectionsStorage extends MastraBase {
  abstract init(): Promise<void>;
  abstract get(key): Promise<StorageToolProviderConnection | null>;
  abstract upsert(input): Promise<StorageToolProviderConnection>;
  abstract list(input): Promise<StorageToolProviderConnection[]>;
  abstract delete(input): Promise<void>;
  abstract dangerouslyClearAll(): Promise<void>;
}
```

### Sibling domain conventions — the pattern is overwhelming

Almost every storage domain uses **expressive, domain-prefixed method names**. The intent is immediately readable from the call site without needing to chase the variable name.

**Datasets** (`datasets/base.ts`)
```ts
createDataset, getDatasetById, deleteDataset, listDatasets,
listItems, getItemById, getItemsByVersion, getItemHistory,
createDatasetVersion, listDatasetVersions
```

**Experiments** (`experiments/base.ts`)
```ts
createExperiment, updateExperiment, getExperimentById, listExperiments,
deleteExperiment, addExperimentResult, updateExperimentResult,
getExperimentResultById, listExperimentResults, deleteExperimentResults,
getReviewSummary
```

**Schedules** (`schedules/base.ts`)
```ts
createSchedule, getSchedule, listSchedules, listDueSchedules,
updateSchedule, updateScheduleNextFire, deleteSchedule,
recordTrigger, listTriggers
```

**Background tasks** (`background-tasks/base.ts`)
```ts
createTask, updateTask, getTask, listTasks, deleteTask, deleteTasks,
getRunningCount, getRunningCountByAgent
```

**Scores** (`scores/base.ts`)
```ts
getScoreById, saveScore, listScoresByScorerId, listScoresByRunId,
listScoresByEntityId
```

**Channels** (`channels/base.ts`)
```ts
saveInstallation, getInstallation, getInstallationByAgent,
getInstallationByWebhookId, listInstallations, deleteInstallation,
saveConfig, getConfig, deleteConfig
```

**Favorites** (`favorites/base.ts`)
```ts
favorite, unfavorite, isFavorited, isFavoritedBatch,
listFavoritedIds, deleteFavoritesForEntity
```

The **only exception** is `blobs/base.ts` — bare `put`, `get`, `has`, `delete`, `putMany`, `getMany`. That fits because a blob store is genuinely a generic content-addressable key-value bag.

**Our class** uses bare CRUD verbs (`get`, `upsert`, `list`, `delete`) — but a tool-provider connection is a domain entity with clear semantics, not a generic blob. It should follow the domain-prefixed pattern.

### Why the sibling pattern wins
1. **Disambiguation when several storage stores get plumbed together.** `storage.toolProviderConnections.get(...)` is fine in isolation, but in handlers you often grab multiple stores. Compare:
   ```ts
   // Current
   await store.get({ authorId, providerId, connectionId });
   await store.list({ authorId, providerId });

   // Pattern-aligned
   await store.getConnection({ authorId, providerId, connectionId });
   await store.listConnections({ authorId, providerId });
   ```
2. **Grep-ability.** `store.get(` matches half the codebase. `getConnection(` matches only this domain.
3. **Future expansion.** When we add things like `countConnectionsByAuthor`, `getConnectionUsage`, `findByLabel`, the bare-verb style forces awkward names; the prefixed style stays consistent.
4. **Consistency with the type names.** Inputs are already named with the domain (`StorageUpsertToolProviderConnectionInput`, `StorageListToolProviderConnectionsInput`). The methods should mirror that.

### Recommended rename — match the dominant `verb-Entity[-By-key]` pattern

| current | suggested | rationale |
|---|---|---|
| `get(key: { authorId, providerId, connectionId })` | `getConnectionById(args)` | matches `getDatasetById`, `getScoreById`, `getExperimentById`, `getExperimentResultById` |
| `upsert(input)` | `upsertConnection(input)` | no sibling uses bare `upsert`; closest precedent is `saveScore` — but `upsert` is more honest about idempotent semantics so keep the verb |
| `list(input: { authorId, providerId?, toolkit? })` | `listConnectionsByAuthor(args)` | matches `listScoresByRunId`, `listScoresByEntityId`, `getRunningCountByAgent` |
| `delete(input)` | `deleteConnection(input)` | matches `deleteDataset`, `deleteSchedule`, `deleteTask`, `deleteExperiment` |
| `dangerouslyClearAll()` | keep | already domain-agnostic by convention (`blobs`, `favorites` use it too) |

### Why `getConnectionById` and `listConnectionsByAuthor` over plain `getConnection`/`listConnections`

The whole point of the rename is **expressiveness at the call site**. Compare:

```ts
// Current — opaque
await store.get({ authorId, providerId, connectionId });
await store.list({ authorId, providerId: 'composio' });

// First-pass rename — better but still leaves you reading the args
await store.getConnection({ authorId, providerId, connectionId });
await store.listConnections({ authorId, providerId: 'composio' });

// Pattern-aligned — intent reads from the method name
await store.getConnectionById({ authorId, providerId, connectionId });
await store.listConnectionsByAuthor({ authorId, providerId: 'composio' });
```

`getConnectionById` signals "primary-key lookup". `listConnectionsByAuthor` signals "scoped to one bucket" — which is the security-critical invariant of this domain. Both names match the existing house style and make the intent grep-able.

### Effort
- Rename 5 methods on abstract base + 2 implementations (inmemory + libsql)
- Update server handlers (`tool-providers.ts:271, 396, 541, 618, 682`)
- Update tests (`inmemory.test.ts`, `libsql index.test.ts`, server `tool-providers.test.ts`)
- No public SDK surface change — these are internal storage methods

Small change, mechanical, no behavior change. Worth doing before we ship the storage domain.

### Files
- `packages/core/src/storage/domains/tool-provider-connections/base.ts`
- `packages/core/src/storage/domains/tool-provider-connections/inmemory.ts`
- `stores/libsql/src/storage/domains/tool-provider-connections/index.ts`
- All callers in `packages/server/src/server/handlers/tool-providers.ts`

## Q9: Can the Agent Builder use a custom model gateway?

**Short answer:** Yes — but only if you override the Builder's default model when registering it.

### How it works today
In `packages/editor/src/ee/agent-builder-agent.ts:171`:
```ts
return new Agent<'builder-agent'>({
  ...
  model: 'openai/gpt-5.5',   // hardcoded default
  memory,
  workspace,
  ...(args || {}),            // ← overrides spread AFTER defaults
  id: 'builder-agent',
  name: 'Agent Builder Agent',
  ...
});
```

`createBuilderAgent(args?)` accepts `Partial<AgentConstructorArgs>` and spreads it **after** the defaults, so `model` is overridable:

```ts
import { createBuilderAgent } from '@mastra/editor/ee';

const builderAgent = createBuilderAgent({
  model: 'my-enterprise-gateway/openai/gpt-4o',
});
```

### Custom gateway registration
Mastra has a full custom gateway system. Register on the `Mastra` instance and reference the gateway prefix in the model string:

```ts
import { Mastra } from '@mastra/core';
import { MastraModelGateway } from '@mastra/core/llm';
import { createBuilderAgent } from '@mastra/editor/ee';

class EnterpriseGateway extends MastraModelGateway {
  readonly id = 'enterprise';
  readonly name = 'Enterprise LLM';
  // ... fetchProviders, resolveLanguageModel
}

const mastra = new Mastra({
  gateways: {
    enterprise: new EnterpriseGateway(),
  },
  agents: {
    builderAgent: createBuilderAgent({
      model: 'enterprise/openai/gpt-4o',  // gateway/provider/model
    }),
  },
});
```

The router parses `gateway/provider/model` (3-part) or `provider/model` (2-part fallback to default gateway). See `packages/core/src/llm/model/router.ts` + `gateway-resolver.ts:8-34`.

### What gateways the Builder model string can target
- **Built-in** (no setup): `models.dev`, `mastra`, `netlify`, `azure-openai`
- **Custom**: Anything extending `MastraModelGateway` and registered via `new Mastra({ gateways: { ... } })`
- **Format**: `[gatewayId/]providerId/modelId` (e.g. `azure-openai/gpt-4o-deployment`, `enterprise/openai/gpt-4o`)

### Caveat
- The Builder requires **tool calling** (it drives `set-agent-instructions`, `set-agent-tools`, etc.). Whatever model you point it at must support function/tool calling.
- The default `openai/gpt-5.5` was chosen because the Builder's prompt is tuned for it; downgrading to a less capable model may produce worse agent configurations even if it doesn't error.

### Documentation
- The override capability is **not documented** in the Agent Builder docs today (`docs/agent-builder/overview.mdx` only mentions `OPENAI_API_KEY` requirement). Worth adding a "Using a custom model" subsection if we want this to be discoverable.

### Files
- `packages/editor/src/ee/agent-builder-agent.ts:37-178` — `createBuilderAgent(args?)` factory
- `packages/core/src/llm/model/gateways/base.ts` — `MastraModelGateway` abstract base
- `packages/core/src/llm/model/router.ts` — model string resolver
- `packages/core/src/llm/model/gateway-resolver.ts:8-34` — `parseModelRouterId`
- `docs/agent-builder/overview.mdx` — currently silent on model override
