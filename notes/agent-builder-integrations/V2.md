# Agent Builder Integrations — V2 Plan

**Status:** planning / research
**Scope:** provider-agnostic integration layer for the agent builder (Composio, Arcade.dev, future providers)
**Stance:** prototypes on `yj/composio-...` and `yj/arcade-agent-builder` are exploratory. V2 starts fresh with one shared abstraction.

---

## 1. Why a reset

Two parallel prototypes shipped (Composio in this repo, Arcade in the sibling repo). Each grew its own:

- Provider-specific provider class (`ComposioToolProvider`, `ArcadeToolProvider`)
- Provider-named UI (`AccountPicker` + `BindingModeToggle` + `ComposioConnectRequiredBadge` vs `IntegrationToolsSection` + `ArcadeConnectRequiredBadge`)
- Provider-named server routes (`/editor/builder/composio/*` vs `/tool-providers/*`)
- Provider-named storage keys (`connectionsByToolkit` vs `bindings`)

But the **shape is the same**: catalog → toolkits → tools → per-toolkit bindings with a kind + identity → fan-out at runtime → connect-required marker mid-chat.

V2 collapses both onto one `IntegrationProvider` interface, one storage shape, one UI, one set of server routes.

---

## 2. v1 scope (author-only)

In:
- Browse catalog of toolkits + tools from any enabled provider
- Add tools to an agent
- Pin **author-mode** credentials at author time (multi-account supported)
- Required label per binding
- Health/status of the integration wiring

Reserved in schema but **UI-hidden in v1**:
- `kind: 'invoker'` — end-user OAuth at runtime
- `kind: 'platform'` — service / shared account
- Per-tool overrides (everything is per-toolkit in v1)

Explicitly out of v1:
- Agent-level `authMode` field (collapsed — per-binding `kind` is sufficient)
- Mixed mode within one agent (no UI; storage allows it for future)
- Mid-chat connect badge (lives with invoker mode in v1.5)

## 3. Default architecture

### 3.1 Provider interface

```ts
interface IntegrationProvider {
  readonly id: string;                     // 'composio' | 'arcade'
  readonly displayName: string;

  // Catalog (read-only, allowlist-filtered by FilteredToolProvider wrapper)
  listToolkits(): Promise<Toolkit[]>;
  listTools(toolkit: string): Promise<ToolDescriptor[]>;

  // Runtime resolution — author-mode v1: userId comes from binding.userId
  resolveTools(opts: ResolveToolsOpts): Promise<Record<string, MastraTool>>;

  // OAuth surface (required; throw NotSupportedError if a provider lacks one)
  authorize(opts: AuthorizeOpts):
    Promise<{ url: string; authId: string }>;
  getAuthStatus(authId: string):
    Promise<'pending' | 'completed' | 'failed'>;
  getConnectionStatus(opts: { items: Array<{ bucketId: string; toolkit: string }> }):
    Promise<Record<string, { connected: boolean }>>;     // keyed by bucketId

  // Health
  getHealth(): Promise<ProviderHealth>;
}

type AuthorizeOpts = {
  toolkit: string;
  bucketId: string;             // provider-opaque OAuth bucket identifier
  toolName?: string;
};

type ResolveToolsOpts = {
  toolSlugs: string[];
  toolMeta: Record<string, { description?: string }>;
  bucketId: string;             // provider-opaque OAuth bucket identifier
  requestContext?: Record<string, unknown>;
};
```

Why required (not optional) on the interface:
- Arcade already exposes `authorize` / `getAuthStatus` / `getConnectionStatus` on `ToolProvider`.
- Composio has the same semantics under different names (`initiateConnection`, `listConnections`). Renaming aligns the surface.
- A provider that genuinely can't OAuth implements them as `throw new NotSupportedError(...)` and the UI degrades.

### 3.2 Storage shape

```ts
// On stored agent
integrationTools: Record<providerId, ProviderConfig>;

type ProviderConfig = {
  tools: Record<string, { description?: string }>;
  bindings: Record<string, ConnectionBinding[]>;        // key = toolkit slug (v1)
};

type ConnectionBinding = {
  kind: 'author' | 'invoker' | 'platform';   // v1 writes 'author' only
  bucketId: string;            // provider-opaque OAuth bucket identifier
  label: string;               // REQUIRED, used for UI display + tool-name suffix
};
```

Validation:
- `label`: required, non-empty, ≤32 chars, regex `[A-Za-z0-9 _-]+`
- `label`: **case-insensitive unique** within `bindings[toolkit]` — original case preserved for display. `Work` and `work` collide.
- `kind` in v1: always `'author'` on write (UI hides other choices); schema accepts all three for forward-compat
- `bucketId`: required for author/platform; reserved/empty for invoker (resolved at runtime)
- Tool-name suffix derived from `label` (sanitized, uppercased)

Notes:
- **One opaque field**, not two. Both providers conceptually need the same thing — "which OAuth bucket?" — they just spell it differently:
  - **Composio:** `bucketId` = `connectedAccountId` (`ca_xxx`). The provider's own constructor config carries the platform `userId`; the binding's `bucketId` is what the adapter injects into `beforeExecute`.
  - **Arcade:** `bucketId` = `user_id`. Multiple Gmails for the same human = multiple synthetic `user_id`s, each a distinct token bucket.
- Provider-specific knobs (Composio's auth scope, Arcade's verified-email requirement) live **inside the provider adapter**, not in the shared binding shape.
- If a future provider truly needs more than one opaque field, add a single optional `metadata?: Record<string, unknown>` escape hatch — but only when needed.
- v1 has no agent-level `authMode` field. The prototype's `authMode`/`authIdentity` is removed entirely. Per-binding `kind` is sufficient.

#### Schema evolution: additive-only, no version field

V1 deliberately omits a `version` field. Instead the schema follows additive-only rules so old agents keep working across minor releases:

- New fields are always **optional**
- Field semantics never change — new meaning = new field name
- Removing a field is **not allowed** in v1.x
- Unknown fields are **preserved** on read/write (Zod `.passthrough()` at the binding + provider-config level)

```ts
const ConnectionBindingSchema = z.object({
  kind: z.enum(['author', 'invoker', 'platform']),
  bucketId: z.string(),
  label: z.string().min(1).max(32).regex(/^[A-Za-z0-9 _-]+$/),
}).passthrough();
```

This buys forward-compat without committing to a versioning scheme:
- v1.5 adds optional `invokerScope?: 'user' | 'team'` → old agents load fine
- v2 adds optional `toolSlug?: string` for per-tool bindings → old agents load fine
- Old clients reading new agents preserve unknown fields → no data loss on round-trip

If a truly breaking change is needed later (e.g. removing `bucketId`), that's the moment to introduce versioning — and we'll know because the additive rule fails.

**Migration from prototypes:** wipe and re-save. Prototype data is throwaway. V1 schema rejects unknown shape with a clear "please re-save this agent" error — no normalizer code path.

### 3.3 Provider registry

```ts
new Mastra({
  editor: new MastraEditor({
    toolProviders: [
      new FilteredToolProvider({
        inner: new ComposioToolProvider({ apiKey: '...' }),
        allowedTools: ['gmail.*', 'slack.*'],
      }),
      new FilteredToolProvider({
        inner: new ArcadeToolProvider({ apiKey: '...' }),
        allowedTools: ['Gmail.*'],
      }),
    ],
  }),
});
```

- Flat array, not `registries.composio.*` (drops Composio prototype's nested shape).
- `FilteredToolProvider` is the allowlist wrapper from Arcade prototype — generic, provider-neutral.
- Order in the array determines tie-break when two providers expose the same toolkit slug.

### 3.3.1 One provider instance, two surfaces

There is **one** `ComposioToolProvider` (and one `ArcadeToolProvider`) per app, registered once on `MastraEditor`. Both the agent builder UI and the runtime fan-out call into the **same instance**.

```
Mastra
  └── editor: { toolProviders: [composio, arcade] }
                          ▲                ▲
                          │                │
            ┌─────────────┘                └──────────────┐
            │                                             │
   Agent builder (author-time)              Runtime (request-time)
   - listToolkits / listTools                - resolveTools (fan-out)
   - authorize / auth-status                 - per-binding userId + connectionRef
   - connection-status                       - reads bindings from stored agent
```

This is unchanged from today's Composio integration — there is **not** a separate "agent builder" provider and "editor" provider. The same `ToolProvider` instance is reused.

What V2 changes:
- Drops the EE `registries.composio.*` config (the agent-builder-specific gating layer in `packages/core/src/agent-builder/ee/`) and folds allowlisting into the generic `FilteredToolProvider` wrapper.
- Renames provider-specific types (`ResolvedComposioRegistry`, `ConnectionPin`, etc.) to provider-neutral equivalents in a shared `tool-provider/` module.
- Keeps the single-instance pattern.

### 3.4 Server routes

Single neutral surface for both providers:

```
GET  /api/tool-providers                            list configured providers
GET  /api/tool-providers/:id/toolkits               list allowed toolkits
GET  /api/tool-providers/:id/tools?toolkit=...      list allowed tools
POST /api/tool-providers/:id/authorize              start OAuth, returns { url, authId }
GET  /api/tool-providers/:id/auth-status/:authId    poll auth flow
GET  /api/tool-providers/:id/connection-status      per-toolkit connection state
GET  /api/tool-providers/:id/health                 provider health
```

All accept `userId` from the client and forward to the provider. Server is intentionally dumb about provider-specific semantics.

### 3.5 Runtime fan-out

`resolveStoredIntegrationTools(integrationTools, requestContext)` is generic — no provider branches.

```
for each providerId in integrationTools:
  cfg = integrationTools[providerId]
  provider = registry.get(providerId)

  for each toolkit in cfg.bindings:
    bindings = cfg.bindings[toolkit]
    toolSlugs = tools in cfg.tools whose slug belongs to toolkit
    skipSuffix = bindings.length === 1

    for binding in bindings:
      resolved = provider.resolveTools({
        toolSlugs,
        toolMeta: cfg.tools,
        bucketId: binding.bucketId,
        requestContext,
      })

      suffix = skipSuffix ? '' : '__' + buildBindingSuffix(binding, usedSuffixes)
      for [slug, tool] in resolved:
        desc = tool.description
        if !skipSuffix:
          desc = `${desc}\n\nRoutes through connection: ${binding.label}`
        out[slug + suffix] = { ...tool, id: slug + suffix, description: desc }
```

`buildBindingSuffix` is lifted from Arcade prototype: `__LABEL`, collision → `_2`, etc. v1 always has a label, so `__INVOKER` / `__HASH` fallbacks aren't needed yet but stay in the helper for v1.5.

**Provider responsibility ends at `resolveTools` for a single `(userId, connectionRef?)` pair.** The provider does **not** know about fan-out, suffixes, or LLM-facing descriptions. This keeps providers small and ensures rename/hint logic lives in one place.

### 3.5.1 LLM-facing shape

The LLM sees **one tool per `(toolSlug × binding)` pair**, never the raw bindings.

Storage:
```ts
bindings: {
  gmail: [
    { kind: 'author', label: 'Work',     bucketId: 'ca_work' },     // Composio: ca_xxx
    { kind: 'author', label: 'Personal', bucketId: 'ca_home' },
  ],
}
tools: { 'gmail.fetch_emails': {}, 'gmail.send_email': {} }
```

For Arcade the same shape, different bucket meaning:
```ts
bindings: {
  Gmail: [
    { kind: 'author', label: 'Work',     bucketId: 'mastra:foo:gmail:work' },
    { kind: 'author', label: 'Personal', bucketId: 'mastra:foo:gmail:personal' },
  ],
}
```

LLM-visible tools after fan-out (4 entries):
```
gmail.fetch_emails__WORK
  Fetch emails from Gmail.
  Routes through connection: Work

gmail.fetch_emails__PERSONAL
  Fetch emails from Gmail.
  Routes through connection: Personal

gmail.send_email__WORK   ...
gmail.send_email__PERSONAL   ...
```

Key invariants:
- The provider (Composio / Arcade) **never sees `__WORK`** — that suffix lives only inside Mastra's tool registry. The tool's `execute` closure carries the binding's `bucketId`.
- Single-binding toolkits skip the suffix entirely (LLM sees natural slug).
- The appended description line is what the LLM actually routes on; the suffix is the dispatch key.
- Bad labels (`account1`, `account2`) → bad LLM routing. Required + unique label enforcement is load-bearing.

### 3.6 Connect-required marker (deferred to v1.5)

When invoker mode lands, the unified tool-result marker is:

```json
{
  "error": true,
  "__connectRequired": true,
  "providerId": "composio",
  "toolkit": "gmail",
  "userId": "...",
  "message": "..."
}
```

One detector in `tool-fallback.tsx`, one `ConnectRequiredBadge` component, no per-provider variants.

## 4. UI

### 4.1 Author-time tools panel

- Single "Tools" panel in the agent builder (no provider tabs).
- Add-tools dialog lists toolkits across all configured providers; tool rows show a provider chip.
- Each toolkit row has an inline `AccountPicker` (multi-select), **co-located with the toolkit's tool list** — no separate Tools page, no modal/drawer.
  - "+ Add account" → mints a `bucketId`, opens OAuth popup
  - Each row has required `label` input
  - Inline validation: label required, label unique within toolkit
- **Empty state, not placeholder row:** when a toolkit has no bindings yet, the row shows a "⚠ No accounts connected yet" hint with an "+ Add account" button. We do **not** insert a placeholder binding row with empty `bucketId`/`label` — that would violate the schema.
- Form save is blocked while any toolkit-with-tools has zero bindings (validation surface).
- No mode toggle in v1 (all author).

### 4.2 Authorize flow

```
Picker → onAuthorize(toolkit, bucketId)
  → POST /tool-providers/:id/authorize { toolkit, bucketId }
  → { url, authId }
  → open popup
  → poll GET /auth-status/:authId until 'completed'
  → refresh connection-status (batched), mark row "Authorized"
```

**Re-authorize semantics** (token expired or revoked):

- Picker row in `disconnected` state shows a "Reauthorize" button instead of a fresh "+ Add account"
- Clicking it calls `authorize()` with the **existing** `bucketId` → token refreshes in place
- Both Composio (`ca_xxx` re-auth) and Arcade (same `user_id` re-auth) do this natively at the SDK level
- The binding row is unchanged on the agent — same `bucketId`, same `label`, fresh token
- This avoids orphaning any other agents that share the same `bucketId`

**Switching accounts** (different identity, same toolkit slot) is *not* re-auth:

- Trash the existing binding → "+ Add account" → mints a new `bucketId`
- Forces the user's intent to be explicit; no silent re-pointing

### 4.3 Health pill

Per-agent pill in the Tools panel header, summarizing provider status:

```
[ Composio ✓ ] [ Arcade ⚠ ]
  click ↓
  Gmail     ✓ 2 connected
  Slack     ⚠ 1 of 2 connected — "Personal" disconnected
  Calendar  ✕ 0 connected
```

- One batched `getConnectionStatus(items)` call per provider on panel open. `items` = every `(bucketId, toolkit)` pair currently bound on the agent.
- Top-level chip shows provider-level rollup (`✓` all connected / `⚠` partial / `✕` none).
- Popover row per toolkit, showing `n of m connected` and which labels are disconnected.
- Each disconnected row has a "Reauthorize" link that routes through §4.2's flow.

Cost is bounded: one call per provider, batched across all bindings — independent of agent size.

### 4.4 What we drop from the prototypes

- `BindingModeToggle` (no kind toggle in v1)
- `ComposioConnectRequiredBadge` + `ArcadeConnectRequiredBadge` (re-added in v1.5 as one generic badge)
- `useComposioConnectBridge`, `useComposioConnections`, `connect-link-modal` (replaced by generic `useToolProviderAuth` + generic modal)
- `authMode` radio (deleted, not just hidden)
- Memory `resourceId` per-mode switch (deferred with invoker mode)

## 5. agentBuilderTool schema

LLM-facing tool that lets the builder agent add integration tools to an agent.

```ts
integrations: {
  add?: Array<{
    providerId: 'composio' | 'arcade';
    toolSlug: string;                  // e.g. 'gmail.fetch_emails' or 'Gmail.ListEmails'
    // bindings written by humans in the UI — builder agent only adds tools,
    // not credentials. New toolkit gets a placeholder binding that the user
    // must visit Tools page to authorize.
  }>;
  remove?: Array<{ providerId: string; toolSlug: string }>;
}
```

- Builder agent never writes `label` or `userId` — those are human-only inputs.
- Placeholder binding (`{ kind: 'author', label: '<toolkit>', userId: '' }`) is added when a toolkit is first referenced and bindings are empty; UI surfaces "needs auth" until the human authorizes.

## 6. What carries over from the prototypes

Concrete pieces to lift:

- `ToolProvider` registry pattern (`mastra.editor.toolProviders[]`, flat)
- `ResolveToolProviderToolsOptions` shape — extended with `connectionRef`
- `FilteredToolProvider` allowlist + fan-out wrapper (from Arcade)
- Multi-account fan-out + rename + injected pin (Composio variant: via `beforeExecute`; Arcade variant: via `user_id` per execute)
- Tool-name suffix algorithm (`buildBindingSuffix`)
- Account picker UX (label + authorize button)
- Health helper + pill
- Server route shape (`/tool-providers/*`)
- Storage-as-JSON via `StorageMCPClientToolsConfig`

What gets redesigned:

- Naming throughout — provider-neutral
- Storage key: `integrationTools[providerId]` (drop `connectionsByToolkit`)
- Binding shape: single opaque `bucketId`; require `label`
- Connect-required machinery: deferred, unified when re-added
- `agentBuilderTool` schema — provider-neutral
- Drop `authMode`/`authIdentity` entirely

## 7. Provider mapping cheatsheet

| Concept | Composio | Arcade |
|---|---|---|
| `bucketId` interpretation | `connectedAccountId` (`ca_xxx`) | `user_id` (often verified email) |
| Auth scope source | Provider constructor (`apiKey` + platform userId) | Implicit per `user_id` |
| Multi-account per human | multiple `ca_xxx` under same Composio userId | multiple distinct `user_id`s (one per account) |
| Per-call routing | `connectedAccountId` injected via `beforeExecute` | `user_id` passed to `tools.execute` |
| Authorize entrypoint | `initiateConnection` → URL | `client.tools.authorize` → URL |
| Auth poll | `listConnections` filter by user | `client.auth.status(authId)` |
| Authorize-on-demand | requires explicit call | `authorization_url` in tool error |

V1 handles both with **one binding shape**, one runtime fan-out, and one opaque `bucketId` per binding. Each provider's adapter knows how to interpret its own `bucketId`.

## 8. Phasing

### v1 — Author mode only
- Provider interface + registry
- Generic server routes
- Generic storage shape + Zod schemas
- Generic UI (Tools panel, picker, authorize flow)
- Composio + Arcade providers implement the interface
- agentBuilderTool integrations branch
- Health helper + pill

### v1.5 — Invoker mode
- Re-introduce `kind: 'invoker'` in UI
- Unified `__connectRequired` marker + `ConnectRequiredBadge`
- Memory `resourceId` switch (author → `agentId`; invoker → `currentUser.id`)
- Auto-retry pattern after OAuth completes

### v2 — Platform mode + per-tool overrides
- Admin-gated `kind: 'platform'`
- Per-tool bindings layered on top of per-toolkit (key changes from `toolkit` to `toolSlug` for overrides)
- Workspace-level connection pool

## 9. Open questions (smaller, post-defaults)

### Provider API
- ~~Q1. `getConnectionStatus` granularity~~ → **resolved**: batch. Accepts `items: Array<{ bucketId, toolkit }>`, returns `Record<bucketId, { connected }>`. UI checks N bindings in one call.
- ~~Q2. `resolveTools` granularity~~ → **resolved**: one binding at a time (per-binding). Runtime owns fan-out; provider stays dumb. See §3.5 and §3.5.1.
- ~~Q3. `connectionRef` typing~~ → **resolved**: collapsed `userId` + `connectionRef` into a single opaque `bucketId`. Both providers conceptually need the same primitive — "which OAuth bucket?" — just spelled differently. Provider-specific knobs live in the adapter, not the shared shape. Escape hatch: add optional `metadata?: Record<string, unknown>` only if a future provider truly needs it.

### UI
- ~~Q4. Account picker location~~ → **resolved**: inline per toolkit in the Tools panel. No separate Tools page, no modal/drawer. See §4.1.
- ~~Q5. Placeholder binding visibility~~ → **resolved**: empty-state row (no placeholder). Form save blocked while any toolkit-with-tools has zero bindings. See §4.1.

### Storage / migration
- ~~Q7. Migration from prototype agents~~ → **resolved**: wipe and re-save. Prototype data is throwaway. V1 rejects unknown shape with a clear error.
- ~~Q8. Schema version field~~ → **resolved**: no version field. Additive-only schema with Zod `.passthrough()` preserves unknown fields. Versioning revisited only on a truly breaking change. See §3.2.

### Validation
- ~~Q9. Label uniqueness case-sensitivity~~ → **resolved**: case-insensitive uniqueness, preserve original case for display. See §3.2.
- ~~Q10. Reserved labels~~ → **deferred**: revisit when invoker/platform modes ship.

### Auth
- ~~Q11. Re-auth same `(human, toolkit)`~~ → **resolved**: reuse the existing `bucketId`, refresh the token in place. Both SDKs do this natively. Switching to a different account is an explicit delete + add — not a re-auth. See §4.2.
- ~~Q12. Token TTL / revocation surfacing~~ → **resolved**: lazy detection. Pre-flight `getConnectionStatus(items)` on Tools-panel open marks revoked rows "Disconnected — Reauthorize". No proactive polling. Runtime failures fall back to provider error in v1, unified `__connectRequired` in v1.5.

### Runtime
- ~~Q13. Revoked token mid-runtime~~ → **resolved**: v1 surfaces the provider error verbatim (author-only — the author owns the connection and sees the failure). v1.5 maps to unified `__connectRequired` marker → inline Connect badge. Same flow as "no connection yet".
- ~~Q14. Health pill scope~~ → **resolved**: per-agent pill with per-provider summary, per-toolkit breakdown in the popover. One batched `getConnectionStatus` call per provider on panel open. See §4.3.
