# CostGuardProcessor

The `CostGuardProcessor` monitors cumulative token usage and estimated cost across the agentic loop, blocking or warning when configurable limits are exceeded.

It uses `processInputStep` to check limits before each LLM call. For `resource` and `thread` scopes, it queries the observability storage APIs to retrieve cumulative usage and cost data across runs.

Supports three scoping modes:

- **Run scope** (default): Tracks usage within a single agent run
- **Resource scope**: Tracks cumulative usage per `resourceId` across runs (requires observability storage)
- **Thread scope**: Tracks cumulative usage per `threadId` across runs (requires observability storage)

## Usage example

Track token usage within a single agent run:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  limits: { maxTotalTokens: 100_000 },
})
```

Track cumulative cost per thread across runs:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  limits: { maxCost: 1.0 },
  scope: 'thread',
})
```

Attach to an agent with an `onViolation` callback:

```typescript
import { Agent } from '@mastra/core/agent'
import { CostGuardProcessor } from '@mastra/core/processors'

const agent = new Agent({
  name: 'my-agent',
  model: 'openai/gpt-5-nano',
  processors: {
    input: [
      new CostGuardProcessor({
        limits: { maxCost: 5.0 },
        scope: 'resource',
        onViolation: ({ limitType, usage, limit, scopeKey }) => {
          console.log(`Limit ${limitType} exceeded for ${scopeKey}: ${usage}/${limit}`)
        },
      }),
    ],
  },
})
```

## Error behavior

When the `block` strategy is active (default), `CostGuardProcessor` calls `abort()` with `retry: false` when any limit is exceeded. The TripWire metadata includes:

- `processorId`: `'cost-guard'`
- `usage`: Current cumulative usage (`inputTokens`, `outputTokens`, `totalTokens`, `estimatedCost`, `costUnit`)
- `limit`: The configured limits object
- `scope`: The active scope (`'run'`, `'resource'`, or `'thread'`)
- `scopeKey`: The scope identifier for resource/thread scopes (if applicable)

## Scoping behavior

| Scope | Tracks across runs | Requires | Requires context |
| - | - | - | - |
| `run` | No | Nothing | No |
| `resource` | Yes | Observability storage with `getMetricAggregate` | `resourceId` in `RequestContext` |
| `thread` | Yes | Observability storage with `getMetricAggregate` | `threadId` in `RequestContext` |

For `resource` and `thread` scopes, the processor queries the observability storage APIs (`getMetricAggregate`) to retrieve cumulative token usage metrics (`mastra_model_total_input_tokens`, `mastra_model_total_output_tokens`) along with associated `estimatedCost` data. If the Mastra instance does not have observability storage configured with `getMetricAggregate` support, an error is thrown at registration time.

If the required context ID is missing at runtime, the processor falls back to run-scoped tracking. Observability query failures are handled with a fail-open strategy: if a query fails, historical usage starts from zero.
