# CostGuardProcessor

The `CostGuardProcessor` enforces monetary cost limits across the agentic loop, blocking or warning when a configurable cost threshold is exceeded.

It uses `processInputStep` to check the cost limit before each LLM call. For `resource` and `thread` scopes, it queries the observability storage APIs to retrieve cumulative cost data across runs within a configurable time window (defaults to 7 days).

For token-based limits, use `TokenLimiterProcessor` instead.

Supports three scoping modes:

- **Run scope** (default): Tracks cost within a single agent run
- **Resource scope**: Tracks cumulative cost per `resourceId` across runs (requires observability storage)
- **Thread scope**: Tracks cumulative cost per `threadId` across runs (requires observability storage)

## Usage example

Track cost within a single agent run:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  maxCost: 1.0,
})
```

Track cumulative cost per thread with a 24-hour window:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  maxCost: 5.0,
  scope: 'thread',
  window: '24h',
})
```

Attach to an agent with an `onViolation` callback:

```typescript
import { Agent } from '@mastra/core/agent'
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  maxCost: 5.0,
  scope: 'resource',
  window: '30d',
})

costGuard.onViolation = ({ detail }) => {
  console.log(`Cost exceeded for ${detail.scopeKey}: $${detail.usage}/$${detail.limit}`)
}

const agent = new Agent({
  name: 'my-agent',
  model: 'openai/gpt-5-nano',
  processors: {
    input: [costGuard],
  },
})
```

## Error behavior

When the `block` strategy is active (default), `CostGuardProcessor` calls `abort()` with `retry: false` when the cost limit is exceeded. The TripWire metadata includes:

- `processorId`: `'cost-guard'`
- `usage`: Current cumulative usage (`estimatedCost`, `costUnit`)
- `maxCost`: The configured cost limit
- `scope`: The active scope (`'run'`, `'resource'`, or `'thread'`)
- `scopeKey`: The scope identifier for resource/thread scopes (if applicable)

## Scoping behavior

| Scope | Tracks across runs | Requires | Requires context |
| - | - | - | - |
| `run` | No | Nothing | No |
| `resource` | Yes | Observability storage with `getMetricAggregate` | `resourceId` in `RequestContext` |
| `thread` | Yes | Observability storage with `getMetricAggregate` | `threadId` in `RequestContext` |

For `resource` and `thread` scopes, the processor queries the observability storage APIs (`getMetricAggregate`) with a time window filter to retrieve cumulative cost data. If the Mastra instance does not have observability storage configured with `getMetricAggregate` support, an error is thrown at registration time.

If the required context ID is missing at runtime, the processor falls back to run-scoped tracking. Observability query failures are handled with a fail-open strategy: if a query fails, historical cost starts from zero.
