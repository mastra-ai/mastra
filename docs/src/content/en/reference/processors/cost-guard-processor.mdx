# CostGuardProcessor

The `CostGuardProcessor` monitors cumulative token usage and step count across the agentic loop, blocking or warning when configurable limits are exceeded.

It uses `processInputStep` to check limits before each LLM call, and `processOutputStep` to record usage after each LLM response for cross-run scopes.

Supports three scoping modes:

- **Run scope** (default): Tracks usage within a single agent run
- **Resource scope**: Tracks cumulative usage per `resourceId` across runs (requires server cache)
- **Thread scope**: Tracks cumulative usage per `threadId` across runs (requires server cache)

## Usage example

Track token usage within a single agent run:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  limits: { maxTotalTokens: 100_000, maxSteps: 10 },
})
```

Track cumulative usage per thread across runs:

```typescript
import { CostGuardProcessor } from '@mastra/core/processors'

const costGuard = new CostGuardProcessor({
  limits: { maxTotalTokens: 500_000 },
  scope: 'thread',
})
```

Attach to an agent:

```typescript
import { Agent } from '@mastra/core/agent'
import { CostGuardProcessor } from '@mastra/core/processors'

const agent = new Agent({
  name: 'my-agent',
  model: 'openai/gpt-5-nano',
  processors: {
    input: [
      new CostGuardProcessor({
        limits: { maxTotalTokens: 50_000, maxSteps: 5 },
      }),
    ],
  },
})
```

## Error behavior

When the `block` strategy is active (default), `CostGuardProcessor` throws a `TripWire` error with `retry: false` when any limit is exceeded. The TripWire metadata includes:

- `processorId`: `'cost-guard'`
- `usage`: Current cumulative usage (`inputTokens`, `outputTokens`, `totalTokens`, `steps`)
- `limit`: The configured limits object
- `scope`: The active scope (`'run'`, `'resource'`, or `'thread'`)
- `scopeKey`: The cache key used for resource/thread scopes (if applicable)

## Scoping behavior

| Scope | Tracks across runs | Requires cache | Requires context |
| - | - | - | - |
| `run` | No | No | No |
| `resource` | Yes | Yes | `resourceId` in `RequestContext` |
| `thread` | Yes | Yes | `threadId` in `RequestContext` |

For `resource` and `thread` scopes, the processor reads and writes usage data via the Mastra server cache. If the cache is unavailable or the required context ID is missing, the processor falls back to run-scoped tracking.

Cache failures are handled with a fail-open strategy: if a cache read fails, usage starts from zero; if a cache write fails, the step continues without persisting.
