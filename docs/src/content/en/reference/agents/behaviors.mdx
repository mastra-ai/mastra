# Behaviors API

The `@mastra/behaviors` package provides mutable path-based behavior definitions, durable per-thread state, state-signal projection, tool policy enforcement, and periodic transition recovery.

## `BehaviorResolver`

```typescript
interface BehaviorResolver {
  readonly id: string
  readonly root: '$root'
  resolve(id: BehaviorPath): Promise<BehaviorNode | undefined>
  children(id: BehaviorPath): Promise<readonly BehaviorNode[]>
  parent(id: BehaviorPath): BehaviorPath | undefined
}
```

A resolver reads the current node and its immediate destinations. It doesn't need to construct a complete graph.

### `BehaviorNode`

| Property | Type | Description |
| - | - | - |
| `id` | `BehaviorPath` | Canonical path beginning with `$root`. |
| `name` | `string` | Short name accepted by `behavior({ name })`. |
| `version` | `string` | Version used to detect node changes. |
| `instructions` | `string` | Instructions shown to the acting agent. |
| `judgeInstructions` | `string` | Instructions supplied only to the transition judge. |
| `skills` | `readonly string[]` | Skill paths loaded for the active node. |
| `tools` | `readonly string[]` | Tool names allowed by the active node. |
| `model` | `string` | Optional acting-model identifier. |
| `judgeModel` | `string` | Optional judge-model identifier. |
| `guards` | `readonly BehaviorGuard[]` | Deterministic checks before navigation. |
| `judge` | `boolean` | Whether navigation requires a judge. |
| `destinations` | `readonly string[]` | Additional absolute or relative destinations. |
| `periodic` | `{ intervalMs, destination }` | Optional scheduled navigation. |

## `InMemoryBehaviorResolver`

A mutable TypeScript resolver.

```typescript
const resolver = new InMemoryBehaviorResolver('debug', rootNode)
resolver.set('$root/behaviors/investigate', investigateNode)
resolver.remove('$root/behaviors/investigate')
```

`set()` inserts or replaces a node. `remove()` removes the path and all descendants. `children()` discovers immediate direct children and nodes under a `behaviors/` container.

## `FileSystemBehaviorResolver`

```typescript
const resolver = await FileSystemBehaviorResolver.create('./behaviors/debug', 'debug')
```

Each node directory contains `BEHAVIOR.md`. The Markdown body is `instructions`; YAML frontmatter accepts `version`, `tools`, `skills`, `model`, `judgeModel`, `judge`, `guards`, `destinations`, and `periodic`. An optional sibling `JUDGE.md` supplies `judgeInstructions`.

The resolver discovers child directories on each `children()` call and reads node files on each `resolve()` call. Path traversal, escaping symlinks, absolute skill paths, and skills outside the behavior root are rejected.

`loadBehaviorDirectory(directory, id?)` is a convenience wrapper around `FileSystemBehaviorResolver.create()`.

## `BehaviorSignalProvider`

```typescript
new BehaviorSignalProvider({
  resolver,
  store,
  resolveThreadId,
  resolveModel,
  resolveSkillInstructions,
  judgeIntent,
  unavailableModel: 'error',
})
```

The provider contributes:

- A state processor that projects the current behavior path and destinations.
- A routing processor for active-node instructions, skills, and model.
- An output processor and execution wrappers that enforce intent and allowed tools.
- Stable `behavior` and `behavior_intent` tools.
- Optional scheduler lifecycle management.

The provider initializes `resolver.root` for a thread when behavior state is first needed.

## `BehaviorTransitionEngine`

```typescript
await engine.transition({
  threadId,
  name: '$root/behaviors/write-test',
})
```

`name` can be an available node's canonical path or short name. The engine resolves availability from `children()` plus `parent()`, evaluates current-node guards before the judge, checks the latest runtime revision, commits atomically, clears intent, and records transition history. Idempotency keys are generated internally unless an infrastructure caller supplies one.

## Runtime stores

Both stores implement `BehaviorRuntimeStore`:

- `InMemoryBehaviorRuntimeStore` serializes transactions in one process.
- `LibSQLBehaviorRuntimeStore` persists records and serializes competing writers through LibSQL transactions and retry handling.

`transactThread()` is the authority for state changes. `listDue()` supports scheduler recovery after restart.

## `BehaviorScheduler`

`BehaviorScheduler` polls `listDue()`, claims work with a persisted lease, and calls the transition engine with the active node's periodic destination. It records completion checkpoints, applies retry backoff, rejects stale results, and releases leases after errors or judge timeouts.

Call `start()` once for a provider lifecycle and `stop()` before replacing or disposing the provider.

## State signal

The state processor emits `<current-behavior>` with:

- Behavior ID.
- Canonical current path.
- Acting instructions.
- Available child, parent, and declared destination paths.
- Current intent.

The projection excludes judge instructions. Its cache key changes only when model-facing node data, destinations, or intent changes, which avoids repeated snapshots for internal runtime revisions.
