# GithubSignals

`GithubSignals` connects Mastra agent threads to GitHub pull request events. It exposes a `processor` for agent configuration, helper signals for subscribe and unsubscribe actions, and an idle-time poller that sends PR notifications back to the thread after agent runs finish.

Use it when an agent needs CI failure, PR comment, or review notifications while it continues working in a thread.

## Usage example

Create one `GithubSignals` controller, add its processor to the agent, then register the agent with the controller.

```typescript title="src/mastra/index.ts"
import { Agent } from '@mastra/core/agent'
import { GithubSignals, ghSignals } from '@mastra/core/signals'

const github = new GithubSignals({
  repo: 'mastra-ai/mastra',
})

const agent = new Agent({
  id: 'code-agent',
  name: 'Code agent',
  instructions: 'Help with code review and CI failures.',
  model,
  inputProcessors: [github.processor],
})

github.addAgent(agent)

await github.init({ memory, resourceId: 'user_123' })

await agent.sendSignal(ghSignals.prSubscribe({ prNumber: 1234 }), {
  resourceId: 'user_123',
  threadId: 'thread_456',
})
```

Call `init()` during application startup to explicitly rehydrate persisted subscriptions. The subscribe signal tells the model that the thread is subscribed to the PR. On the next input step, the processor records the subscription in thread metadata and snapshots the current PR state as a silent baseline. After each agent run finishes, it polls GitHub once with lightweight `gh api` REST calls. Existing checks, comments, and reviews are not replayed when the subscription starts.

## Signal helpers

Import `ghSignals` from `@mastra/core/signals`.

### `ghSignals.prSubscribe(input)`

Creates a signal that subscribes the current thread to a GitHub pull request.

```typescript
await agent.sendSignal(ghSignals.prSubscribe({ prNumber: 1234, repo: 'mastra-ai/mastra' }), {
  resourceId: 'user_123',
  threadId: 'thread_456',
})
```

Returns: `CreatedAgentSignal`.

### `ghSignals.prUnsubscribe(input)`

Creates a signal that unsubscribes the current thread from a GitHub pull request.

```typescript
await agent.sendSignal(ghSignals.prUnsubscribe({ prNumber: 1234, repo: 'mastra-ai/mastra' }), {
  resourceId: 'user_123',
  threadId: 'thread_456',
})
```

Returns: `CreatedAgentSignal`.

### `ghSignals.prNotification(input)`

Creates an internal notification signal for poller output. Most applications do not need to call this helper directly.

```typescript
const signal = ghSignals.prNotification({
  prNumber: 1234,
  repo: 'mastra-ai/mastra',
  kind: 'review',
  title: 'GitHub review',
  details: 'test2',
  user: 'TylerBarnes',
  reviewState: 'COMMENTED',
})
```

Returns: `CreatedAgentSignal`.

Notification signals use compact GitHub-specific reminder types such as `github-ci-failure`, `github-comment`, `github-review`, and `github-command-error`. Metadata like PR number, repo, user, review state, URL, and failed check count is stored on signal attributes/metadata while the message body stays concise for model context.

Comment and review notifications are authorization-gated. By default, only users with `admin`, `maintain`, or `write` repository permissions can trigger those notifications. Bot users must be explicitly included in `authorizedBots`.

When the processor detects likely PR work in recent shell/tool activity, such as `gh pr` commands or `git push`, and the thread is not subscribed yet, it emits a one-time `github-subscription-hint` reminder asking the agent to use the `github` subscribe tool. MastraCode also shows active PR subscriptions as a sky-blue `PR#123` badge next to the thread title in the status line.

## Methods

### Agent registration

#### `addAgent(agent, options?)`

Registers an agent so the poller can send notification signals to it.

```typescript
github.addAgent(agent)
```

Returns: `GithubSignals`.

#### `removeAgent(agentOrId)`

Removes an agent and stops polling subscriptions owned by that agent.

```typescript
github.removeAgent('code-agent')
```

### Subscription lifecycle

#### `init(options)`

Explicitly rehydrates persisted subscriptions from memory. Call this during application startup after registering agents.

```typescript
await github.init({ memory, resourceId: 'user_123' })
```

Returns: `Promise<GithubPRSubscriptionMetadata[]>`.

#### `start(subscriptions?)`

Adds explicit subscriptions for future idle-time polls.

```typescript
github.start([
  {
    agentId: 'code-agent',
    resourceId: 'user_123',
    threadId: 'thread_456',
    repo: 'mastra-ai/mastra',
    prNumber: 1234,
    createdAt: new Date().toISOString(),
    updatedAt: new Date().toISOString(),
  },
])
```

Returns: `GithubSignals`.

#### `addSubscription(subscription)`

Adds one subscription for future idle-time polls.

```typescript
github.addSubscription({
  agentId: 'code-agent',
  resourceId: 'user_123',
  threadId: 'thread_456',
  repo: 'mastra-ai/mastra',
  prNumber: 1234,
  createdAt: new Date().toISOString(),
  updatedAt: new Date().toISOString(),
})
```

#### `removeSubscription(subscription)`

Removes one active subscription.

```typescript
github.removeSubscription({
  agentId: 'code-agent',
  resourceId: 'user_123',
  threadId: 'thread_456',
  repo: 'mastra-ai/mastra',
  prNumber: 1234,
})
```

### Polling

#### `markActive(context)` / `markIdle(context)`

Marks a thread as actively running or idle. The processor calls these lifecycle hooks automatically. Active threads are skipped, and `markIdle()` triggers one immediate poll for that thread.

```typescript
github.markActive({ agentId: 'code-agent', resourceId: 'user_123', threadId: 'thread_456' })
await github.markIdle({ agentId: 'code-agent', resourceId: 'user_123', threadId: 'thread_456' })
```

#### `poll(context?)`

Runs one REST poll cycle for idle subscriptions. Pass a thread context to poll only that thread.

```typescript
await github.poll()
```

#### `destroy()`

Clears registered agents, active subscriptions, and active-thread state.

```typescript
github.destroy()
```

## Thread metadata

The processor stores subscription state under `thread.metadata.mastra.githubSignals`. The metadata includes processed signal IDs and active subscriptions keyed by repository and PR number. Each subscription also stores watermarks for failed checks, latest comment timestamp, and latest review timestamp so the first poll after subscribing only reports future changes.

## GitHub CLI requirements

The default command runner uses `gh api` REST endpoints with `execFile`. It reads PR head metadata, issue comments, PR reviews, commit check runs, and collaborator permissions. The environment running the agent must have the GitHub CLI installed and authenticated. Provide `commandRunner` to use a custom GitHub client or to mock GitHub in tests.
