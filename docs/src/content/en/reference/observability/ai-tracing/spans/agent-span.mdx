# Agent Spans

Agent spans track the execution of AI agents, including their interactions with LLMs, tool usage, and memory operations.

## Span Types

### AGENT_RUN

Represents the complete execution of an agent from start to finish.

#### Attributes

<PropertiesTable
  props={[
    {
      name: "agentId",
      type: "string",
      description: "Unique identifier for the agent",
      required: true,
    },
    {
      name: "agentName",
      type: "string",
      description: "Human-readable agent name",
      required: false,
    },
    {
      name: "instructions",
      type: "string",
      description: "System instructions provided to the agent",
      required: false,
    },
    {
      name: "availableTools",
      type: "string[]",
      description: "List of tool names available to the agent",
      required: false,
    },
    {
      name: "maxSteps",
      type: "number",
      description: "Maximum number of reasoning steps allowed",
      required: false,
    },
    {
      name: "temperature",
      type: "number",
      description: "Temperature setting for LLM responses",
      required: false,
    },
  ]}
/>

#### Example

```typescript
{
  type: 'AGENT_RUN',
  name: 'customer-support-agent',
  startTime: new Date(),
  attributes: {
    agentId: 'agent_123',
    agentName: 'Customer Support Agent',
    instructions: 'Help customers with their inquiries',
    availableTools: ['search-kb', 'create-ticket'],
    maxSteps: 10,
    temperature: 0.7
  },
  input: {
    message: 'I need help with my order',
    threadId: 'thread_456'
  }
}
```

```typescript
  input: {
    messages: [
      { role: 'system', content: 'You are a helpful assistant' },

  output: {
    content: 'I cannot provide real-time weather information...',
    finishReason: 'stop'

  metadata: {
    promptTokens: 25,
    completionTokens: 50,
    totalTokens: 75
  }
}
```

## Common Patterns

### Nested Span Structure

Agent operations typically create a hierarchy of spans:

```
AGENT_RUN
├── LLM_GENERATION (initial response)
├── TOOL_CALL (search for information)
├── LLM_GENERATION (process tool results)
└── TOOL_CALL (take action)
```

### Memory Operations

Memory recall and storage are tracked as tool calls:

```typescript
{
  type: 'TOOL_CALL',
  name: 'memory-recall',
  attributes: {
    toolId: 'memory_semantic_recall',
    toolName: 'Semantic Memory Recall',
    toolType: 'memory'
  },
  input: {
    query: 'previous orders',
    topK: 5,
    threshold: 0.7
  },
  output: {
    memories: [...],
    count: 3
  }
}
```

### Error Tracking

Failed operations include error details:

```typescript
{
  type: 'TOOL_CALL',
  name: 'api-call',
  status: 'ERROR',
  attributes: {
    toolId: 'external_api',
    toolName: 'External API',
    success: false,
    errorMessage: 'Connection timeout'
  },
  error: {
    name: 'NetworkError',
    message: 'Connection timeout after 30s',
    stack: '...'
  }
}
```

## Metadata Best Practices

### Token Tracking

Always include token counts for LLM operations:

```typescript
metadata: {
  promptTokens: 150,
  completionTokens: 200,
  totalTokens: 350,
  estimatedCost: 0.0105 // Optional cost tracking
}
```

### Performance Metrics

Include timing and performance data:

```typescript
metadata: {
  latency: 1250, // ms
  cacheHit: false,
  retryCount: 0,
  queueTime: 50 // ms
}
```

### Context Information

Add relevant context for debugging:

```typescript
metadata: {
  sessionId: 'session_789',
  userId: 'user_123',
  environment: 'production',
  feature: 'chat-support',
  version: '1.2.0'
}
```
