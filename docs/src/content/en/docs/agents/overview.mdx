# Agents overview

Agents use LLMs and tools to solve open-ended tasks. They reason about goals, decide which tools to use, retain conversation memory, and iterate internally until the model emits a final answer or an optional stop condition is met. Agents produce structured responses you can render in your UI or process programmatically. Use agents directly or compose them into workflows or agent networks.

## When to use agents

Use agents when the task is open-ended and the steps aren't known in advance. An agent decides which tools to call, how many times to loop, and when to stop. You provide the goal and constraints instead of defining each step. For predetermined, multi-step processes with explicit control flow, use [workflows](/docs/workflows/overview) instead.

:::tip

Watch an introduction to agents, and how they compare to workflows on [YouTube (7 minutes)](https://youtu.be/0jg2g3sNvgw).

:::

## Quickstart

Create an agent by instantiating the `Agent` class from `@mastra/core` and provide the required properties: `id`, `name`, `instructions`, and `model`.

```typescript title="src/mastra/agents/test-agent.ts"
import { Agent } from '@mastra/core/agent'

export const testAgent = new Agent({
  id: 'test-agent',
  name: 'Test Agent',
  instructions: 'You are a helpful assistant.',
  model: 'openai/gpt-5.4',
})
```

Specify a unique `id` for your agent and provide a descriptive `name`.

The `instructions` define the agent's behavior, personality, and capabilities. They're system-level prompts that establish the agent's core identity and expertise. The `model` is specified as `provider/model-name` using Mastra's [model router](/models).

To make the agent available throughout your application, register it in your Mastra instance (typically located in `src/mastra/index.ts`):

```typescript {5} title="src/mastra/index.ts"
import { Mastra } from '@mastra/core'
import { testAgent } from './agents/test-agent'

export const mastra = new Mastra({
  agents: { testAgent },
})
```

Once registered, it can be called from workflows, tools, or other agents, and has access to shared resources such as memory, logging, and observability features.

Use [Studio](/docs/getting-started/studio) to test your agent with different messages, inspect tool calls and responses, and debug agent behavior.

:::note

Visit the [agent reference](/reference/agents/agent) for more information on available properties and configurations.

:::

## Use your agent

After registration, retrieve your agent with [`mastra.getAgentById()`](/reference/core/getAgentById). Call `.generate()` for a complete response or `.stream()` to deliver tokens in real time. You can call agents from workflow steps, tools, the Mastra Client, or the command line.

<Tabs>
  <TabItem value="generate" label="Generate">
    Returns the full response after all tool calls and steps complete. The result includes `text`, `toolCalls`, `toolResults`, `steps`, and token `usage` statistics.

    ```ts
    const agent = mastra.getAgentById('test-agent')
    const response = await agent.generate('Help me organize my day')
    console.log(response.text)
    ```
  </TabItem>

  <TabItem value="stream" label="Stream">
    Returns a stream you can consume as tokens arrive. The result exposes `textStream` for incremental output and promises for `toolCalls`, `toolResults`, `steps`, and token `usage` that resolve when the stream finishes.

    ```ts
    const agent = mastra.getAgentById('test-agent')
    const stream = await agent.stream('Help me organize my day')

    for await (const chunk of stream.textStream) {
      process.stdout.write(chunk)
    }
    ```
  </TabItem>
</Tabs>

:::note

`mastra.getAgentById()` is preferred over a direct import, since it provides access to the Mastra instance configuration (logger, telemetry, storage, registered agents, etc.).

:::

## Multi-agent systems

A multi-agent system uses multiple agents to solve a task that is too broad or too specialized for a single agent. Instead of building one agent with dozens of tools and a long instruction set, you split responsibilities across focused agents and let a coordinator bring results together.

### When to use multiple agents

Reach for a multi-agent system when:

- A task spans distinct domains (research, writing, data analysis) that benefit from separate instructions and tool sets.
- You want to limit each agent's scope so its context stays small and its responses stay accurate.
- Different parts of a task need different models, memory configurations, or guardrails.

For tasks that a single agent can handle with a few tools, adding more agents introduces unnecessary latency and complexity.

### Supervisor agents

[Supervisor agents](/docs/agents/supervisor-agents) are the recommended way to build multi-agent systems in Mastra. A supervisor agent owns a set of subagents. During execution, the supervisor's LLM decides when to delegate to a subagent, what prompt to send, and how to combine results. Subagents are automatically exposed as callable tools.

Each subagent runs with its own thread and resource ID. This means memory stays isolated: subagents receive the conversation context they need but store only their own delegation prompt and response. The supervisor's memory is unaffected.

You can control delegation behavior with hooks (`onDelegationStart`, `onDelegationComplete`), filter which messages are forwarded (`messageFilter`), and score task completion (`taskCompletionCriteria`). See [Supervisor agents](/docs/agents/supervisor-agents) for a full walkthrough with code examples.

:::tip

Follow the [Build a research coordinator with supervisor agents](/guides/guide/research-coordinator) guide for a step-by-step walkthrough.

:::
