# Response caching

Response caching skips the LLM call and replays a previously cached response when an agent receives an identical request. Use it to drop latency to single-digit milliseconds and avoid paying for repeated calls.

## When to use response caching

Reach for it when the same request shape repeats across users or sessions, for example prompt templates, suggested-prompt buttons, agentic search re-asks, or guardrail LLMs that classify the same input over and over. Skip it when calls trigger external side effects through tools, since cache hits replay tool calls without re-executing them.

## Quickstart

Pass `responseCache: true` on a single call to use the agent's default cache configuration:

```typescript title="src/example.ts"
const result = await agent.generate('What is the capital of France?', {
  responseCache: true,
})
```

The first call runs the LLM normally and writes the response to the cache. Subsequent identical calls return the cached response without invoking the LLM.

## Enable for the whole agent

Set `responseCache` on the agent to cache every call by default:

```typescript title="src/mastra/agents/search-agent.ts"
import { Agent } from '@mastra/core/agent'

export const searchAgent = new Agent({
  name: 'Search Agent',
  instructions: 'You answer questions concisely.',
  model: 'openai/gpt-5',
  responseCache: { ttl: 600 }, // 10 minutes
})
```

Per-call options override agent-level defaults. Pass `responseCache: false` to opt out for a single request.

## Configure TTL, scope, key, and bust

```typescript title="src/example.ts"
await agent.stream(prompt, {
  responseCache: {
    key, // override the auto-derived cache key
    ttl, // per-entry TTL in seconds
    scope, // tenant/user scope, or null to disable scoping
    cache, // custom MastraCache implementation
    bust, // when true, bypass any existing entry and overwrite
  },
})
```

By default, the cache key includes the resource id from `memory` if provided, so two users with separate memory resources never see each other's cached responses. Override with `scope` for explicit tenant isolation:

```typescript title="src/example.ts"
await agent.stream(prompt, {
  responseCache: { scope: ctx.organizationId },
})
```

Pass `scope: null` to deliberately share entries across all callers — only use this for known-public, non-personalized content.

## Provide a custom cache backend

If you don't pass `cache`, the agent uses the cache registered on its `Mastra` instance via `new Mastra({ cache })`. Pass any `MastraCache` implementation to override:

```typescript title="src/mastra/agents/cached-agent.ts"
import { Agent } from '@mastra/core/agent'
import type { MastraCache } from '@mastra/core/cache'

const cache: MastraCache = {
  async get(key) {
    /* ... */
  },
  async set(key, value, ttlSeconds) {
    /* ... */
  },
}

export const agent = new Agent({
  name: 'Cached Agent',
  instructions: '...',
  model: 'openai/gpt-5',
  responseCache: { cache },
})
```

## How caching is implemented

Response caching runs as a `ResponseCache` input processor that hooks into the new `processLLMRequest` and `processLLMResponse` hooks. When `responseCache` is enabled (per-call or on the agent), Mastra auto-registers the processor for you unless you have already added a `ResponseCache` to your `inputProcessors`. You can also wire the processor manually if you need fine-grained control over its position in the pipeline.

This means the cache key is derived from the _resolved_ `LanguageModelV2Prompt` Mastra is about to send to the model — i.e. _after_ memory has loaded and earlier input processors have run — and each step in an agentic tool loop is independently cached.

## What's in the cache key

When you don't supply `key`, Mastra derives one deterministically from the inputs that change the LLM's response at this step: `agentId`, `stepNumber` (so each step in a tool loop has its own cache entry), `scope`, model identity (`provider`, `modelId`, spec version), and the resolved `prompt` (post-memory + post-processors). Any change to these inputs automatically invalidates the cache.

### Customize the cache key

Pass `key` as a function to derive your own cache key from any subset of those inputs. The function receives the same inputs Mastra would have hashed and returns a string (or a `Promise<string>`):

```typescript title="src/example.ts"
import { buildAgentResponseCacheKey } from '@mastra/core/agent'

await agent.stream(input, {
  responseCache: {
    // Cache only on the model id and the resolved prompt tail — ignore
    // step number, scope, etc.
    key: ({ model, prompt }) => `qa:${model.modelId}:${JSON.stringify(prompt).slice(-200)}`,
  },
})

// Or reuse Mastra's hashing helper while overriding individual fields:
await agent.stream(input, {
  responseCache: {
    key: inputs => buildAgentResponseCacheKey({ ...inputs, scope: 'global' }),
  },
})
```

If the function throws, Mastra falls back to the default key derivation so the call still benefits from caching.

## How cache hits work

When the processor finds a cache hit, it short-circuits the LLM call by returning the cached chunks from `processLLMRequest`. The agentic loop synthesizes a stream from those chunks instead of calling the model. `agent.generate()` collects them into a `FullOutput`; `agent.stream()` returns a `MastraModelOutput` whose chunks come from the cached buffer, so consumers iterating `fullStream` or awaiting `text`, `usage`, and `finishReason` see the cached values.

Cache writes happen after the response completes. Failed runs (errors, tripwire activations) are not cached, so the next call retries cleanly.
