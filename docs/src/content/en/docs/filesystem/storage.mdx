# Storage

Mastra sets the project's default [storage](/docs/memory/storage) from a `storage.ts` file directly under `src/mastra/`. The file default-exports a store, which replaces the built-in in-memory store used for memory, workflows, observability, and other storage domains.

Without `storage.ts` or a code-registered store, Mastra falls back to in-memory storage. That default is useful for quick local tests, but data is lost on restart and isn't safe for production.

## When to set storage

Add `storage.ts` when your project needs durable state beyond one process lifetime.

| Need | Why storage matters |
| - | - |
| Production persistence | Threads, workflow snapshots, traces, scores, and other records survive restarts. |
| Multi-instance deployments | Every process reads and writes through the same database. |
| Long-running workflows | Suspended and resumed runs need durable workflow snapshots. |
| Local development with history | A file-backed database keeps memory and workflow state between `mastra dev` restarts. |

See [Memory storage](/docs/memory/storage) for setup patterns and [Storage reference](/reference/storage/overview) for domain and schema details.

## Quickstart

Use [`LibSQLStore`](/reference/storage/libsql) for a local file-backed store:

```typescript title="src/mastra/storage.ts"
import { LibSQLStore } from '@mastra/libsql'

export default new LibSQLStore({
  id: 'mastra-storage',
  url: 'file:./mastra.db',
})
```

Mastra discovers the file during `mastra dev` or `mastra build` and registers the store before file-based agents and workflows are registered. This lets storage-dependent primitives bind to the file-based store instead of the default in-memory store.

## Production backends

`storage.ts` can export any Mastra storage adapter, such as LibSQL, PostgreSQL, or MongoDB. Keep this page focused on the filesystem convention, then use [Memory storage](/docs/memory/storage) for setup patterns and the [storage reference](/reference/storage/overview) to inspect adapter schemas and domain support.

For production, use a backend shared by every deployed instance. For local development, a file-backed LibSQL database is usually the smallest durable setup.

## Precedence with code

Code-registered storage wins. If you pass `storage` to [`new Mastra()`](/reference/core/mastra-class), the file-based store is skipped and a warning is logged.

Use code registration when storage setup depends on runtime wiring in `src/mastra/index.ts`. Use `storage.ts` when one project-wide store is enough and you want it discovered by convention.
