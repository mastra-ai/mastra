# Tools

A file-based agent discovers tools from `.ts` and `.js` files directly under its `tools/` directory. Each discovered file is imported at build time, the file must default-export a [`createTool()`](/reference/tools/create-tool) result, and the filename without its extension becomes the tool key the model can call.

Use tools for actions the model can choose during a turn: calling APIs, reading application data, calculating values, or performing side effects.

## When to use tools

Add a tool when the agent needs a callable capability with structured inputs and runtime logic. Use [`instructions.md`](/docs/filesystem/instructions) for always-on behavior guidance and [`skills/`](/docs/filesystem/skills) for on-demand procedures or reference material.

## Quickstart

Place one tool per file under `tools/`. This file is exposed to the agent as `get_weather`.

```typescript title="src/mastra/agents/weather/tools/get_weather.ts"
import { createTool } from '@mastra/core/tools'
import { z } from 'zod'

export default createTool({
  id: 'get_weather',
  description: 'Get the current weather for a city.',
  inputSchema: z.object({
    city: z.string(),
  }),
  outputSchema: z.object({
    city: z.string(),
    tempC: z.number(),
  }),
  execute: async ({ context }) => {
    return { city: context.city, tempC: 21 }
  },
})
```

## Organizing tools

Use one file per tool. Name the file after the action the model should take:

- Use `get_weather.ts`, `search_docs.ts`, or `create_ticket.ts`.
- Avoid vague names like `helper.ts`, `api.ts`, or `utils.ts`.
- Co-locate tests next to the tool as `*.test.ts` or `*.spec.ts`; discovery ignores those files.
- Keep shared helper code outside `tools/` or in files that don't match the discovered extensions under `tools/`.

Because nested directories aren't discovered as tools, group related tools with clear filename prefixes instead of subfolders, such as `calendar_create_event.ts` and `calendar_list_events.ts`.

## Writing good tool descriptions

The `description` field is routing text for the model. The model reads it to decide whether to call the tool, so describe when to use the tool and what result it returns.

Vague:

```typescript
createTool({
  id: 'weather',
  description: 'Weather tool',
})
```

Specific:

```typescript
createTool({
  id: 'get_weather',
  description: 'Get the current temperature and conditions for a city.',
})
```

Prefer descriptions that answer these questions:

- What action does the tool perform?
- When should the model call it?
- What data does it need?
- What does it return?

## Runtime boundary

Tool code runs in your app/server runtime when the agent calls the tool. If a tool needs filesystem or shell isolation, call [workspace](/docs/filesystem/workspace) or sandbox APIs explicitly.

## Call an external API

The following tool calls an external weather API and returns a typed result. It throws when the API request fails so the agent receives a tool error instead of misleading data.

```typescript title="src/mastra/agents/weather/tools/get_weather.ts"
import { createTool } from '@mastra/core/tools'
import { z } from 'zod'

export default createTool({
  id: 'get_weather',
  description: 'Get the current temperature and conditions for a city.',
  inputSchema: z.object({
    city: z.string().describe('City name, such as Austin or Tokyo'),
  }),
  outputSchema: z.object({
    city: z.string(),
    tempC: z.number(),
    condition: z.string(),
  }),
  execute: async ({ context }) => {
    const url = new URL('https://api.example.com/weather')
    url.searchParams.set('city', context.city)

    const response = await fetch(url)

    if (!response.ok) {
      throw new Error(`Weather API request failed: ${response.status}`)
    }

    const data = (await response.json()) as {
      temperature_c: number
      condition: string
    }

    return {
      city: context.city,
      tempC: data.temperature_c,
      condition: data.condition,
    }
  },
})
```

## Shape tool output for the model

Use `toModelOutput` when the model should see a different shape than the raw value returned from `execute`. For example, keep structured data for your application while sending concise text back to the model.

```typescript title="src/mastra/agents/weather/tools/get_weather.ts"
import { createTool } from '@mastra/core/tools'
import { z } from 'zod'

export default createTool({
  id: 'get_weather',
  description: 'Get the current temperature and conditions for a city.',
  inputSchema: z.object({ city: z.string() }),
  execute: async ({ context }) => ({
    city: context.city,
    tempC: 21,
    condition: 'sunny',
  }),
  toModelOutput: output => ({
    type: 'text',
    value: `${output.city}: ${output.tempC}°C and ${output.condition}`,
  }),
})
```

See [`toModelOutput`](/reference/tools/create-tool#parameters) in the `createTool()` reference for the supported output shapes.

## Require approval

Set `requireApproval: true` for tools that need human confirmation before execution, such as sending email, deleting data, or making purchases.

```typescript title="src/mastra/agents/support/tools/refund_order.ts"
import { createTool } from '@mastra/core/tools'
import { z } from 'zod'

export default createTool({
  id: 'refund_order',
  description: 'Refund a customer order after a human approves the action.',
  inputSchema: z.object({ orderId: z.string() }),
  requireApproval: true,
  execute: async ({ context }) => {
    return { orderId: context.orderId, refunded: true }
  },
})
```

See [tool approval](/docs/agent-controller/tool-approvals) for the approval flow and [`requireApproval`](/reference/tools/create-tool#parameters) for the tool option.

## Precedence with config

Discovered tools merge with any `tools` in [`config.ts`](/docs/filesystem/config). On a key collision, `config.tools` wins and a warning is logged.

If `config.tools` is a function, discovered tools are ignored with a warning because function-valued tools can't be statically merged.
