# Server overview

Mastra runs as an HTTP server that exposes your agents, workflows, and other functionality as API endpoints. The server handles request routing, middleware execution, authentication, and streaming responses.

:::info

This page covers the [`server`](/reference/configuration#server-options) configuration options passed to the `Mastra` constructor. For running Mastra with your own HTTP server (Hono, Express, etc.), visit [Server Adapters](/docs/server/server-adapters).

:::

## Server architecture

Mastra uses [Hono](https://hono.dev) as its underlying HTTP server framework. When you build a Mastra application using `mastra build`, it generates a Hono-based HTTP server in the `.mastra` directory.

The server provides:

- API endpoints for all registered agents and workflows
- Custom API routes and middleware
- Authentication across providers
- Request context for dynamic configuration
- Stream data redaction for secure responses

## Configuration

Configure the server by passing a `server` object to the `Mastra` constructor:

```typescript title="src/mastra/index.ts"
import { Mastra } from '@mastra/core'

export const mastra = new Mastra({
  server: {
    port: 3000, // Defaults to PORT env var or 4111
    host: '0.0.0.0', // Defaults to MASTRA_HOST env var or 'localhost'
  },
})
```

:::info

Visit the [configuration reference](/reference/configuration#server-options) for a full list of available server options.

:::

## Server features

- **[Middleware](/docs/server/middleware)**: Intercept requests for authentication, logging, CORS, or injecting request-specific context.
- **[Custom API Routes](/docs/server/custom-api-routes)**: Extend the server with your own HTTP endpoints that have access to the Mastra instance.
- **[Request Context](/docs/server/request-context)**: Pass request-specific values to agents, tools, and workflows based on runtime conditions.
- **[Server Adapters](/docs/server/server-adapters)**: Run Mastra with Express, Hono, or your own HTTP server instead of the generated server.
- **[Custom Adapters](/docs/server/custom-adapters)**: Build adapters for frameworks not officially supported.
- **[Mastra Client SDK](/docs/server/mastra-client)**: Type-safe client for calling agents, workflows, and tools from browser and server environments.
- **[Authentication](/docs/server/auth)**: Secure endpoints with JWT, Clerk, Supabase, Firebase, Auth0, or WorkOS.

## OpenAI-compatible responses API

Mastra also exposes a Responses API-compatible route. This route is an adapter layer on top of Mastra agents, memory, and storage, not a raw provider proxy. By default, the generated server serves it at `/api/v1/responses`, along with `GET /api/v1/responses/:responseId`, `DELETE /api/v1/responses/:responseId`, `POST /api/v1/conversations`, `GET /api/v1/conversations/:conversationId`, `DELETE /api/v1/conversations/:conversationId`, and `GET /api/v1/conversations/:conversationId/items`.

Use a Mastra model string such as `openai/gpt-5` for `model`, and pass `agent_id` to select the registered Mastra agent that should handle the request. Initial Responses requests must target an agent directly. Stored follow-up turns can omit `agent_id` when continuing with `previous_response_id`:

```typescript title="client.ts"
const response = await fetch('http://localhost:4111/api/v1/responses', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    model: 'openai/gpt-5',
    agent_id: 'support-agent',
    input: 'Summarize this ticket for me',
    store: true,
  }),
})
```

Set `stream: true` to receive SSE events compatible with the Responses API event model. Mastra emits the OpenAI-style lifecycle events `response.created`, `response.in_progress`, `response.output_item.added`, `response.content_part.added`, `response.output_text.delta`, `response.output_text.done`, `response.content_part.done`, `response.output_item.done`, and `response.completed`, with `sequence_number` values for deterministic client-side ordering. This compatibility layer only maps the stream Mastra already produces, so it doesn't require extra model calls. Tool-using turns can emit non-message `response.output_item.*` items such as `function_call` alongside assistant message items. `store: true` is only supported for agent-backed requests where the selected agent has memory configured. Stored responses can then be continued with `previous_response_id`. For stored responses, `response.id` is the persisted assistant turn ID in memory, so `GET /api/v1/responses/:responseId` and `DELETE /api/v1/responses/:responseId` operate on that stored assistant turn.

Stored responses also return `conversation_id`. In Mastra, this is the raw memory `threadId`. You can create a conversation explicitly with `POST /api/v1/conversations`, or let the first stored Responses request create it implicitly:

```typescript title="client.ts"
const response = await fetch('http://localhost:4111/api/v1/responses', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    model: 'openai/gpt-5',
    agent_id: 'support-agent',
    input: 'Start a support conversation',
    store: true,
  }),
})

const firstTurn = await response.json()
console.log(firstTurn.conversation_id)
```

If you already have a conversation, pass `conversation_id` on the next request to keep writing to that same thread:

```typescript title="client.ts"
const response = await fetch('http://localhost:4111/api/v1/responses', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
  },
  body: JSON.stringify({
    model: 'openai/gpt-5',
    agent_id: 'support-agent',
    conversation_id: 'thread_123',
    input: 'Continue this conversation',
    store: true,
  }),
})
```

If you pass both `conversation_id` and `previous_response_id`, they must resolve to the same thread. Mastra uses the conversation thread as the source of truth and rejects mismatches.

Configured tool definitions are returned under `tools`, while actual tool activity is emitted through `output` items such as `function_call`, `function_call_output`, and the final assistant `message`. This keeps the Responses adapter aligned with the OpenAI Responses object model while still running through Mastra agents and memory.

For advanced provider-native features, the route also passes `providerOptions` through to the underlying agent `generate()` or `stream()` call. Use this as the escape hatch for provider-specific options that Mastra doesn't normalize at the Responses API layer. When the provider returns provider-native response state, Mastra includes it back on the response object as `providerOptions`, so follow-up requests can reuse values like `openai.responseId`.

The route also supports agents backed by both AI SDK v5+ models and legacy AI SDK v4 models. Responses uses `generate()` and `stream()` for v5+ agents, and automatically falls back to `generateLegacy()` and `streamLegacy()` for legacy agents.

For a working example that keeps the official `openai` SDK and swaps the base URL to Mastra, see the [Responses API demo](https://github.com/mastra-ai/mastra/tree/main/examples/responses-api-demo).

## Stream data redaction

When streaming agent responses, the HTTP layer redacts system prompts, tool definitions, API keys, and similar data from each chunk before sending it to clients. This is enabled by default.

This behavior is only configurable by using [server adapters](/docs/server/server-adapters#stream-data-redaction). For server adapters, stream data redaction is enabled by default, too.

## TypeScript configuration

Mastra requires `module` and `moduleResolution` settings compatible with modern Node.js. Legacy options like `CommonJS` or `node` aren't supported.

```json {4-5} title="tsconfig.json"
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ES2022",
    "moduleResolution": "bundler",
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "strict": true,
    "skipLibCheck": true,
    "noEmit": true,
    "outDir": "dist"
  },
  "include": ["src/**/*"]
}
```
