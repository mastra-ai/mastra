# Behaviors

Behaviors guide an agent through a durable tree of instructions, tools, models, and transition checks. The runtime stores the current behavior path for each thread. A resolver reads only that node and its available destinations, so another process can modify the tree while the agent runs.

Each node has the same contract. `$root` is the initial node, child nodes are discovered recursively, and the parent is always available for upward navigation.

## Define behaviors in TypeScript

Use `InMemoryBehaviorResolver` as a mutable path-indexed collection:

```typescript title="src/mastra/debug-behavior.ts"
import { InMemoryBehaviorResolver } from '@mastra/behaviors'

const defaults = {
  version: '1',
  skills: [],
  guards: [],
  judge: false,
}

export const debugBehavior = new InMemoryBehaviorResolver('debug', {
  ...defaults,
  instructions: 'Understand the issue before editing files.',
  tools: ['read_file', 'run_tests'],
  model: 'openai/gpt-4.1',
})

debugBehavior.set('$root/behaviors/write-test', {
  ...defaults,
  instructions: 'Add focused regression coverage before changing implementation code.',
  tools: ['read_file', 'write_file', 'run_tests'],
})

debugBehavior.set('$root/behaviors/write-test/behaviors/implement', {
  ...defaults,
  instructions: 'Make the smallest implementation change that passes the regression test.',
  tools: ['read_file', 'write_file', 'run_tests'],
})
```

Call `set()` or `remove()` at runtime to update the tree. Active agents resolve the current node again on each step; they don't depend on a frozen global graph.

## Define behaviors on the filesystem

A filesystem tree uses one `BEHAVIOR.md` file per directory. Subdirectories under `behaviors/` become available child behaviors automatically.

```text
behaviors/debug/
├── BEHAVIOR.md
├── JUDGE.md
└── behaviors/
    └── write-test/
        ├── BEHAVIOR.md
        └── behaviors/
            └── implement/
                └── BEHAVIOR.md
```

The Markdown body contains acting instructions. Frontmatter configures the node:

```md title="behaviors/debug/BEHAVIOR.md"
---
tools:
  - read_file
  - run_tests
model: openai/gpt-4.1
judge: true
---

Understand the issue before editing files.
```

`JUDGE.md` contains judge-only instructions and isn't shown to the acting agent. Load the directory as a resolver:

```typescript
import { loadBehaviorDirectory } from '@mastra/behaviors'

const resolver = await loadBehaviorDirectory('./behaviors/debug', 'debug')
```

Filesystem resolvers read `BEHAVIOR.md`, `JUDGE.md`, skills, and child directories when resolving a node. You don't need a manifest or a prebuilt state map.

### Add non-child destinations

Children and the parent are available automatically. Add `destinations` when a node must navigate elsewhere:

```md
---
destinations:
  - $root/shared/verify
---
```

Absolute destinations start with `$root`. A destination must resolve to an existing node before the runtime allows navigation.

## Attach the provider

```typescript
import { Agent } from '@mastra/core/agent'
import { BehaviorSignalProvider, InMemoryBehaviorRuntimeStore } from '@mastra/behaviors'

const provider = new BehaviorSignalProvider({
  resolver: debugBehavior,
  store: new InMemoryBehaviorRuntimeStore(),
  resolveThreadId: requestContext => requestContext?.get('threadId'),
})

const agent = new Agent({
  id: 'debug-agent',
  name: 'Debug agent',
  instructions: 'Follow the active behavior.',
  model: 'openai/gpt-4.1-mini',
  signals: [provider],
})
```

The provider initializes `$root` silently for a new thread. Its state signal shows the current path, acting instructions, available destinations, and current intent.

## Navigate and set intent

The provider exposes two stable tools:

- `behavior({ name })` navigates to an available path or node name. Idempotency keys remain internal to the runtime.
- `behavior_intent({ intent })` sets the intent required by governed tool calls.

The tool names and schemas remain stable across transitions. State-specific policy changes which calls are allowed instead of adding or removing tools.

Before a governed tool runs, the processor checks that its intent matches the active node. A node name is approved directly. Free-form intent can use an intent judge. Rejected calls return corrective feedback to the agent loop and are also blocked by execution-time wrappers.

## Judge transitions

A node can define deterministic `guards`, enable `judge`, and provide `JUDGE.md`. Guards run before the judge. The runtime commits only if the stored revision still matches the state the judge reviewed; stale results fail closed.

## Run periodic work

Set `periodic` in node frontmatter:

```yaml
periodic:
  intervalMs: 60000
  destination: $root/shared/refresh
```

`BehaviorScheduler` leases due work through the behavior store, retries failures with backoff, and recovers persisted due work after restart.

## Persist behavior state

Use `InMemoryBehaviorRuntimeStore` for local sessions. Use `LibSQLBehaviorRuntimeStore` when behavior path, intent, transition history, and scheduler metadata must survive restarts. The behavior store is authoritative; state signals are model-facing projections.
