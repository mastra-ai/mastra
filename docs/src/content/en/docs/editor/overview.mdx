# Editor overview

The editor is a CMS-style system that separates agent configuration from code. Subject-matter experts, prompt engineers, and product teams can iterate on agents directly — changing instructions, swapping tools, and tuning behavior — while developers keep the codebase stable.

Every change is versioned. You can roll back to any previous configuration, target a specific version per request, compare what changed between snapshots, and run A/B tests by routing different users to different versions. The [Studio](/docs/studio/overview) gives you a visual interface for all of this, and you can do everything programmatically through the server API and Client SDK.

## When to use the editor

Use the editor when you want to:

- **Let non-developers iterate** — give subject-matter experts and prompt engineers a way to tune agent behavior without touching code or waiting for deploys.
- **Version everything** — every save creates a snapshot so you can compare changes, roll back instantly, and audit what changed and when.
- **Run experiments** — route different users or requests to different agent versions for A/B testing, canary rollouts, or prompt experimentation.
- **Target specific versions** — pin a version per request, per user, or per environment so production stays stable while new versions are tested.
- **Manage tools at runtime** — add integration tools from Composio or Arcade, or connect MCP servers, without updating code.
- **Override code agents** — change the instructions, tools, or variables of a code-defined agent while keeping the original code as the baseline.

For building agents entirely in code, see the [Agents overview](/docs/agents/overview).

## Get started

Install `@mastra/editor` and pass a `MastraEditor` instance to your Mastra configuration:

```typescript title="src/mastra/index.ts"
import { Mastra } from '@mastra/core'
import { MastraEditor } from '@mastra/editor'

export const mastra = new Mastra({
  agents: {
    /* your code agents */
  },
  editor: new MastraEditor(),
})
```

Once registered, you can manage agents through the Studio UI or programmatically through the server API and Client SDK.

:::note

See the [MastraEditor reference](/reference/editor/mastra-editor) for all configuration options.

:::

## Key concepts

- **[Agents](/docs/editor/agents)** — Override instructions, tools, and variables on code-defined agents. Non-technical team members can iterate and test without touching source files.
- **[Prompts](/docs/editor/prompts)** — Reusable, versioned instruction templates with template variables and display conditions. Share prompt fragments across agents and version them independently.
- **[Tools](/docs/editor/tools)** — Add tools from integration providers, MCP servers, and override tool descriptions — all without redeploying.

## Versioning

Every time you save changes to an agent or prompt block, a new version snapshot is created. Versions give you a full history of your agent's configuration — you can roll back to any previous state, compare what changed between two snapshots, and target specific versions per request for A/B testing or gradual rollouts.

### Version lifecycle

Each version has one of three statuses:

| Status | Description |
| - | - |
| **Draft** | The latest working copy. Every save creates a new draft version. |
| **Published** | The active version used in production. Only one version can be published at a time. |
| **Archived** | A previous version that is no longer active. You can restore any archived version. |

The typical flow is: edit the draft, test it, then activate it to make it the published version. The previously published version becomes archived so you can restore it if needed. You can do this through the Studio UI or programmatically through the API.

This lifecycle makes it safe to experiment. Non-technical team members can iterate on a draft without affecting production traffic, then publish when ready. If something goes wrong, restoring a previous version is a single API call.

### Studio workflow

1. Open an agent's edit page and make changes.
1. Click **Save** to create a new draft version.
1. Click **Publish** to activate the draft as the published version.

### Server endpoints

Version management is available through the stored agents API:

| Method | Endpoint | Description |
| - | - | - |
| `GET` | `/stored/agents/:agentId/versions` | List all versions. |
| `POST` | `/stored/agents/:agentId/versions` | Create a new version. |
| `GET` | `/stored/agents/:agentId/versions/:versionId` | Get a specific version. |
| `POST` | `/stored/agents/:agentId/versions/:versionId/activate` | Set the active version. |
| `POST` | `/stored/agents/:agentId/versions/:versionId/restore` | Restore a version (creates a new copy). |
| `DELETE` | `/stored/agents/:agentId/versions/:versionId` | Delete a version. |
| `GET` | `/stored/agents/:agentId/versions/compare?from=<id>&to=<id>` | Compare two versions. |

### Client SDK

The `Agent` and `StoredAgent` classes in `@mastra/client-js` have methods for version management:

```typescript
import { MastraClient } from '@mastra/client-js'

const client = new MastraClient()
const agent = client.getAgent('support-agent')

// List all versions
const versions = await agent.listVersions()

// Create a new version
const version = await agent.createVersion({
  changeMessage: 'Updated tone to be more friendly',
})

// Activate a version
await agent.activateVersion(version.id)

// Restore a previous version (creates a new version with the same config)
await agent.restoreVersion(versions.items[2].id)

// Compare two versions
const diff = await agent.compareVersions(versions.items[0].id, versions.items[1].id)
```

| Method | Description |
| - | - |
| `agent.listVersions(params?)` | List versions with optional pagination and sorting. |
| `agent.createVersion(params?)` | Create a version snapshot with an optional change message. |
| `agent.getVersion(versionId)` | Get a specific version by ID. |
| `agent.activateVersion(versionId)` | Set a version as the active published version. |
| `agent.restoreVersion(versionId)` | Restore a version by creating a new copy. |
| `agent.deleteVersion(versionId)` | Delete a version. |
| `agent.compareVersions(fromId, toId)` | Compare two versions and return their differences. |

:::note

See the [Client SDK agents reference](/reference/client-js/agents) for the full API.

:::

### React SDK

In the React SDK, pass an `agentVersionId` through `requestContext` when using the `useChat` hook:

```typescript
import { useChat } from '@mastra/react'

function Chat() {
  const { messages, input, handleInputChange, handleSubmit } = useChat({
    agentId: 'support-agent',
    requestContext: {
      agentVersionId: 'abc-123',
    },
  })

  // ... render chat UI
}
```

### Version targeting and experimentation

Because every version has a unique ID, you can route different requests to different agent configurations. This opens up several patterns:

- **A/B testing** — split traffic between two published versions and compare performance metrics.
- **Canary rollouts** — send a small percentage of requests to a new version before promoting it.
- **Per-user targeting** — pin specific users or accounts to a version while others use the default.
- **Environment separation** — use the draft version in staging and the published version in production.

Pass a `versionId` or `status` when calling the agent — through the Client SDK, server query parameters, or React SDK `requestContext` — and the correct version is loaded automatically.
