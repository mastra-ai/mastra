# A2A overview

Mastra exposes agents through the Agent-to-Agent (A2A) protocol so another agent or service can discover them, send work to them, and track task progress over time. Use A2A when you need one Mastra agent to act as a remote agent for another system.

## When to use A2A

- You need a standard agent card and execution endpoint for another service
- You want one Mastra agent to call another agent across a network boundary
- You need task-oriented streaming, resumable tasks, or callback delivery

## How A2A works in Mastra

In Mastra, A2A usually involves two roles:

- **Client agent**: The caller that discovers an agent card, sends a message, and listens for task updates
- **Remote agent**: The Mastra agent running behind the A2A endpoint

The remote agent is still a normal Mastra agent. It keeps using its own instructions, tools, memory, workflows, and server configuration. A2A changes how that agent is reached, not how the agent is authored.

The flow looks like this:

1. Register a standard Mastra agent in your `Mastra` instance.
1. Start Mastra Server or mount Mastra routes into your own server.
1. Mastra exposes that agent through an agent card at `/.well-known/:agentId/agent-card.json` and an execution endpoint at `/a2a/:agentId`.
1. A client agent or application reads the card, sends messages, and receives task, artifact, and status events.

This makes A2A useful for agent-to-agent calls across process boundaries, across services, or across teams. The client doesn't need direct access to the remote agent's internal code or tools. It only needs the A2A endpoint.
