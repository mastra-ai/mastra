# Running experiments

**Added in:** `@mastra/core@1.4.0`

An experiment runs every item in a dataset through a target (an agent, a workflow, or a scorer) and then optionally scores the outputs. Use a scorer as the target when you want to evaluate an LLM judge itself. Results are persisted to storage so you can compare runs across different prompts, models, or code changes.

## Basic experiment

Call [`startExperiment()`](/reference/datasets/startExperiment) with a target and scorers:

```typescript title="src/mastra/experiments/basic.ts"
import { mastra } from '../index'

const dataset = await mastra.datasets.get({ id: 'translation-dataset-id' })

const summary = await dataset.startExperiment({
  name: 'gpt-5.1-baseline',
  targetType: 'agent',
  targetId: 'translation-agent',
  scorers: ['accuracy', 'fluency'],
})

console.log(summary.status) // 'completed' | 'failed'
console.log(summary.succeededCount) // number of items that ran successfully
console.log(summary.failedCount) // number of items that failed
```

`startExperiment()` blocks until all items finish. For fire-and-forget execution, see [async experiments](#async-experiments).

## Studio

You can also run experiments in [Studio](/docs/studio/overview). After you've added a dataset item, open it and select **Run Experiment** and configure the target, scorers, and options.

After running an experiment, the **Experiments** tab shows all runs for that dataset (with status, counts, and timestamps). Select an experiment to see per-item results, scores, and execution traces.

In the **Experiments** tab, select **Compare** and choose two or more experiments to compare their scores and results side by side.

## Experiment targets

You can point an experiment at a registered agent, workflow, or scorer.

### Registered agent

Point to an agent registered on your Mastra instance:

```typescript title="src/mastra/experiments/agent-target.ts"
const summary = await dataset.startExperiment({
  name: 'agent-v2-eval',
  targetType: 'agent',
  targetId: 'translation-agent',
  scorers: ['accuracy'],
})
```

Each item's `input` is passed directly to `agent.generate()`, so it must be a `string`, `string[]`, or `CoreMessage[]`.

### Registered workflow

Point to a workflow registered on your Mastra instance:

```typescript title="src/mastra/experiments/workflow-target.ts"
const summary = await dataset.startExperiment({
  name: 'workflow-eval',
  targetType: 'workflow',
  targetId: 'translation-workflow',
  scorers: ['accuracy'],
})
```

The workflow receives each item's `input` as its trigger data.

### Registered scorer

Point to a scorer to evaluate an LLM judge against ground truth:

```typescript title="src/mastra/experiments/scorer-target.ts"
const summary = await dataset.startExperiment({
  name: 'judge-accuracy-eval',
  targetType: 'scorer',
  targetId: 'accuracy',
})
```

The scorer receives each item's `input` and `groundTruth`. LLM-based judges can drift over time as underlying models change, so it's important to periodically realign them against known-good labels. A dataset gives you a stable benchmark to detect that drift.

## Replaying tool calls

**Added in:** `@mastra/core@1.42.0`

Agent targets execute their tools live by default. For agents that read or write mutable external state (support tickets, CRM records), this makes experiments non-deterministic: the external resource drifts between runs, and tool writes hit production on every run.

Tool replay freezes the external world instead. The experiment intercepts each tool call and returns the output recorded in a prior traced run, so you can re-run the same items and study how the agent behaves against identical observations. The agent itself stays free to diverge — different reasoning, ordering, or arguments are reported, not prevented.

Recordings come from traces. Any traced agent run records its tool calls as spans, and experiments persist a `traceId` for every item result, so a previous experiment can serve directly as the recording:

```typescript title="src/mastra/experiments/tool-replay.ts"
// 1. Run the experiment live with tracing enabled (the recording)
const baseline = await dataset.startExperiment({
  name: 'support-agent-baseline',
  targetType: 'agent',
  targetId: 'support-agent',
})

// 2. Re-run with tool replay — no live tool executions
const replayed = await dataset.startExperiment({
  name: 'support-agent-replay',
  targetType: 'agent',
  targetId: 'support-agent',
  toolReplay: { fromExperimentId: baseline.experimentId },
})
```

Each item resolves its source trace from the first available of: `replayTraceId` on the item (inline data), `metadata.replayTraceId` (storage-backed items), then the `traceId` recorded for the same item ID in `fromExperimentId`'s results.

Matching is per-tool and ordered: each call to a tool consumes the next recorded event for that tool, so repeated calls to the same tool replay in their original order. Recorded errors are re-thrown so the agent sees the same failure. Recorded arguments are compared for diagnostics but never used for matching.

### Miss policy

When the agent makes a call with no recorded event remaining, the `onMiss` policy decides what happens:

- `'error'` (default): The item fails with a `TOOL_REPLAY_MISS` error. No live tool runs, and the item is not retried.
- `'passthrough'`: The call executes live and is recorded as a miss in the report.

```typescript
const replayed = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'support-agent',
  toolReplay: {
    fromExperimentId: baseline.experimentId,
    onMiss: 'passthrough',
  },
})
```

:::warning

With `'passthrough'`, unmatched calls run against the real world, including writes. Avoid it for agents with side-effecting tools unless every write is recorded.

:::

An item with no recording at all fails with `TOOL_REPLAY_NO_RECORDING` regardless of `onMiss`, and is not retried: replay never runs an item silently live. This also applies when the source trace was purged or not yet flushed. When using `fromExperimentId` with inline data, give items explicit `id`s; auto-generated ids change between runs and never match the prior experiment's results.

### Matching policy

By default, recorded events match the agent's calls per tool, in recorded order (`matching: 'fifo'`): the next recorded event for a tool is served regardless of arguments, and argument drift is reported in `argMismatches` rather than prevented. This is the right mode for evaluating a changed agent — a modified prompt reformulates its arguments by nature, and an exact-match requirement would fail the very thing being measured.

For contract-style tests where any deviation must fail, use `matching: 'strict'`: a recorded event is served only when the call's arguments exactly match the recorded call's arguments (canonicalized comparison — runtime-injected nullish keys are ignored). Anything else is a miss, handled by `onMiss`. With multiple recorded argument variants for the same tool, each variant answers only its exact arguments; `argMismatches` is empty by construction in this mode. The contract cuts both ways: recorded calls the agent never consumed fail the item with `TOOL_REPLAY_UNCONSUMED` (not retried) — an agent that stops calling an expected tool cannot pass silently. Under `'fifo'`, leftover events are only reported in `unconsumed`.

```typescript
const strict = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'support-agent',
  toolReplay: { fromExperimentId: baseline.experimentId, matching: 'strict' },
})
```

The matching policy applies to the whole run. To be lenient about one tool whose arguments drift benignly (a generated SQL string, a free-text search query) while staying strict everywhere else, mock that tool — mocks take precedence over the replay queues per tool, so `toolMocks: { runSql: { output: recordedRows } }` answers every `runSql` call regardless of arguments while the remaining tools stay under the strict contract. A tool answered by a suppressing mock is exempt from the strict unconsumed-contract — its recorded calls can never be consumed, so they stay visible in the report's `unconsumed` as signal without failing the item. Expect-only entries observe live calls and stay inside the contract. The other direction needs no escape hatch: under `'fifo'` nothing fails on drift, and `argMismatches` tells you where it happened.

### Tool mocks

`toolMocks` overrides individual tools without needing a recording — usable with or without `toolReplay`. Mocks take precedence over replay queues per tool. In a mock-only run, unmocked tools execute live; when `toolReplay` is active, an unmocked tool with no recorded event left is a miss handled by `onMiss` — it never silently runs live.

```typescript
const mocked = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'support-agent',
  toolMocks: {
    // Static stub — every call returns this; the tool never executes
    weatherInfo: { output: { temp: 20, condition: 'sunny' } },
    // Failure injection — every call throws; the tool never executes
    paymentApi: { error: { name: 'TimeoutError', message: 'timed out' } },
    // Assertion — the run must call this tool once with these args, or the item fails
    createTicket: { output: { ok: true }, expect: { args: { priority: 'high' }, calledTimes: 1 } },
    // Function mock — replaces execute entirely (code-only; not available over HTTP)
    searchDocs: async ({ input }) => fakeIndex.search(input),
  },
})
```

An entry with only `expect` doesn't stub anything: the call executes normally (replayed or live) and only the assertion is recorded. `expect.calledTimes: 0` asserts the tool must NOT be called. An unsatisfied expectation fails the item with `TOOL_MOCK_EXPECTATION_FAILED` (not retried — the run is over, retrying cannot change it); outcomes are listed in the report's `expectations` array, and per-tool usage in `mocks`. Experiments run with mocks are stamped in their metadata (`mockedTools`) and — like replay runs — are refused as replay sources: mocked calls record only synthetic tool spans, which replay extraction ignores.

### Divergence report

Every replayed item result carries a `toolReplay` report describing how the run differed from the recording:

```typescript
const report = replayed.results[0].toolReplay

console.log(report.replayedCount) // recorded events the agent consumed
console.log(report.misses) // calls with no recorded event
console.log(report.unconsumed) // recorded events the agent never requested
console.log(report.argMismatches) // replayed calls whose args differed from the recording
```

The report is the top-level `toolReplay` field on every result — both on the summary returned by `startExperiment()` and on stored results ([`listExperimentResults()`](/reference/datasets/listExperimentResults) and the HTTP API), where it persists in its own column. Scorers never see it: the output they evaluate stays free of replay metadata, so replay runs score the same way as baseline runs. The report is kept for failed items too, except when the failure happens outside the agent run itself (for example an item timeout racing the run).

Treat divergence as a trust signal for scores. An entry in `argMismatches` means the agent received a re-paired recorded observation: matching is per-tool and ordered, so the replayed output came from a call whose original arguments differed. A `passthrough` miss means a live observation entered the run. In either case, inspect the report before trusting the item's scores.

The report also carries two recording-quality signals: `redactedPayloadCount` counts recorded payloads containing the sensitive-data redaction marker (the agent received `[REDACTED]` values), and `staleRecording` is `true` when the recording came from an older version of the dataset item than the one being replayed.

`toolReplay` is also available through the experiment trigger API: pass it in the body of `POST /api/datasets/:datasetId/experiments` alongside `targetType` and `targetId`.

### What replay freezes — and what it doesn't

Replay freezes the external world _as the recorded run observed it through tools_ — nothing more. Knowing what stays live keeps comparisons honest:

- **The model runs live.** A replay run is a real LLM run making its own decisions against frozen observations — that is the point. Identical scores are not guaranteed even with an unchanged agent; sampling alone can vary the output.
- **The agent definition resolves at run time.** Instructions, model, and tools are whatever the agent is _now_. Pin `agentVersion` in the experiment config to run a stored version, or leave it unpinned when the change is exactly what you are measuring. The pinned version is persisted on the experiment; unpinned runs resolve the current definition and store no version.
- **No conversation state is involved.** Each experiment item executes as a standalone run without a memory thread, so between two experiments nothing can drift. One caveat: an item saved from a production trace replays that run's tool observations, not its conversation — if the original run leaned on thread memory, the replayed agent reasons without it.
- **Item inputs can drift.** `staleRecording` flags an item edited after its recording was made; pin the dataset version (`version`) for stable inputs.
- **`requestContext` passes through as configured.** If the agent reads it, keep it identical between the recording run and the replay run.

For a contract-style comparison, the checklist is: pinned `agentVersion`, pinned dataset `version`, identical `requestContext`, and `matching: 'strict'`.

### Limitations

- Tool replay supports agent targets only. Workflow targets and inline `task` functions are rejected at setup.
- Tools run inside sub-agents (agent-as-tool) or workflow-as-tool internals are not intercepted. The top-level call replays as a single recorded output.
- Provider-executed tools (for example provider-side web search) run on the model provider as part of the live model call. They cannot be intercepted, record no tool spans, and never appear in the divergence report — a run using them is not frozen by replay.
- Tools injected mid-run by input processors (for example tool-search or skill-discovery processors) are not intercepted yet. They execute live during replay, and their recorded events surface as `unconsumed` in the report.
- Replayed and mocked calls do not execute the real tool. They appear in the run's own trace as synthetic tool spans flagged with `metadata.toolReplay.synthetic: true` (plus an `outcome` such as `replayed` or `mocked`, and the consumed recording's `sequence` for replayed calls), so trace timelines stay readable. The spans record the served observation, not an execution; replay extraction ignores them, so they can never be replayed from. The divergence report stays the primary tool for inspecting replay behavior.
- Trajectory scorers extract trajectories from the run's trace. Synthetic tool spans make replay trajectories visible, so baseline and replay trajectories are comparable in shape — but the replay run's tool steps represent served recordings or mocks, not executions (the synthetic flag is carried in each step's metadata). Interpret trajectory scores with that in mind.
- Replay uses per-execution tool hooks, which override agent-level `hooks` (such as a configured `beforeToolCall`) for the replayed runs. With `onMiss: 'passthrough'`, missed calls execute live without those agent-level guards. A tool's own `onInputAvailable` and `onOutput` callbacks still fire for replayed calls.
- Recordings inherit tracing's sensitive-data filtering. Redacted input fields surface as `argMismatches` in the report, and redacted output fields replay as `[REDACTED]` values. If a tool returns sensitive values that replay needs, adjust the `sensitiveDataFilter` option on your observability config.
- Traces are exported asynchronously. When replaying immediately after a recording run in the same process, ensure the tracing exporter has flushed before starting the replay; an incomplete trace replays as missing events.
- Tool spans that never completed (the recorded run crashed, was aborted, or is still running) are skipped during extraction. The affected calls follow the `onMiss` policy instead of replaying empty outputs.
- Tools that require approval are not replayed: the approval gate runs before tool execution, so the run suspends for approval before the replay hook can return the recorded output. Avoid `toolReplay` with approval-required tools.
- A source trace should contain a single top-level agent run. A trace spanning several runs contributes all of their tool calls to the replay queues.
- Replay runs emit only synthetic-flagged tool spans, which extraction skips, so an experiment that was itself replayed cannot serve as a recording. `fromExperimentId` pointing at a replay experiment is rejected at setup; a `replayTraceId` pointing at a replay run's trace finds zero events and every call follows the `onMiss` policy.

## Async experiments

`startExperiment()` blocks until every item completes. For long-running datasets, use [`startExperimentAsync()`](/reference/datasets/startExperimentAsync) to start the experiment in the background:

```typescript title="src/mastra/experiments/async.ts"
const { experimentId, status } = await dataset.startExperimentAsync({
  name: 'large-dataset-run',
  targetType: 'agent',
  targetId: 'translation-agent',
  scorers: ['accuracy'],
})

console.log(experimentId) // UUID
console.log(status) // 'pending'
```

Poll for completion using [`getExperiment()`](/reference/datasets/getExperiment):

```typescript
let experiment = await dataset.getExperiment({ experimentId })

while (experiment.status === 'pending' || experiment.status === 'running') {
  await new Promise(resolve => setTimeout(resolve, 5000))
  experiment = await dataset.getExperiment({ experimentId })
}

console.log(experiment.status) // 'completed' | 'failed'
```

## Configuration options

### Concurrency

Control how many items run in parallel (default: 5):

```typescript
const summary = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'translation-agent',
  maxConcurrency: 10,
})
```

### Timeouts and retries

Set a per-item timeout (in milliseconds) and retry count:

```typescript
const summary = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'translation-agent',
  itemTimeout: 30_000, // 30 seconds per item
  maxRetries: 2, // retry failed items up to 2 times
})
```

Retries use exponential backoff. Abort errors are never retried.

### Aborting an experiment

Pass an `AbortSignal` to cancel a running experiment:

```typescript
const controller = new AbortController()

// Cancel after 60 seconds
setTimeout(() => controller.abort(), 60_000)

const summary = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'translation-agent',
  signal: controller.signal,
})
```

Remaining items are marked as skipped in the summary.

### Pinning a dataset version

Run against a specific snapshot of the dataset:

```typescript
const summary = await dataset.startExperiment({
  targetType: 'agent',
  targetId: 'translation-agent',
  version: 3, // use items from dataset version 3
})
```

## Viewing results

### Listing experiments

```typescript
const { experiments, pagination } = await dataset.listExperiments({
  page: 0,
  perPage: 10,
})

for (const exp of experiments) {
  console.log(`${exp.name} — ${exp.status} (${exp.succeededCount}/${exp.totalItems})`)
}
```

### Experiment details

```typescript
const experiment = await dataset.getExperiment({
  experimentId: 'exp-abc-123',
})

console.log(experiment.status)
console.log(experiment.startedAt)
console.log(experiment.completedAt)
```

### Item-level results

```typescript
const { results, pagination } = await dataset.listExperimentResults({
  experimentId: 'exp-abc-123',
  page: 0,
  perPage: 50,
})

for (const result of results) {
  console.log(result.itemId, result.output, result.error)
}
```

## Understanding the summary

`startExperiment()` returns an `ExperimentSummary` with counts and per-item results:

- `completedWithErrors` is `true` when the experiment finished but some items failed.
- Items cancelled via `signal` appear in `skippedCount`.
